# 应用 Sortlytic UI 设计系统

## 目标

把 Sortlytic 维持为“macOS 原生感、专业数据工作台、暖中性极简、证据导向”的本地优先桌面应用。优先保证真实状态、操作边界、数据可读性与可访问性，再处理装饰。

## 开始前

1. 读取仓库根目录 `AGENTS.md`，把其中当前项目规则视为强制约束。
2. 完整读取 [references/design-system.md](references/design-system.md)，再提出设计或修改代码。
3. 检查相关现有组件与样式；至少检查 `apps/macos/src/index.css`、`App.css`、`App.responsive.css`，并按任务继续检查页面专属 CSS/TSX。
4. 有用户参考图时先检查图形语义和实际资源。有既有打包图标时复用 `apps/macos/src-tauri/icons/icon.png`，禁止重画近似 Logo。
5. 区分旧截图、浏览器预览与当前打包产物。旧截图只提供历史参考，不能覆盖当前源码或用户对当前实机的结论。

## 决策顺序

按以下优先级解决冲突：

1. 用户在当前任务中的明确要求。
2. 当前 `AGENTS.md` 的项目规则。
3. 本技能的设计基线。
4. 当前源码中可复用且未违反前三项的既有模式。
5. 历史截图、旧版界面和演示数据。

不要用偶然存在的样式漂移反向改写设计基线。确需改变基线时，同步更新本技能的仓库版与全局安装版。

## 工作流程

### 1. 确定范围

- 判断任务属于新页面、现有页面改造、组件抽取、主题适配、响应式修复、可访问性修复或实机验收。
- 写代码前说明方案和修改边界；超过 3 个文件时拆成独立修改包。
- 保留用户已有改动，不把无关重构混入 UI 修改包。

### 2. 建立界面层级

- 先确定页面上下文、主标题、说明、主操作和状态反馈。
- 再确定单栏或“主内容 + 检查栏”结构；没有检查栏时不得预留空列。
- 用留白、字号、字重、浅色表面和细分隔建立层级，避免无语义的反相黑块、重复套卡和过度装饰。
- 让任务、证据、成本、来源、限制和导出结果保持可见且可追溯。

### 3. 复用设计系统

- 优先复用 CSS 变量、按钮、状态标签、应用内列表框、任务卡、状态带和现有间距。
- 使用 Lucide 线性图标或真实 SVG/图源，不使用 Emoji 充当图标。
- 新颜色先映射到已有业务语义，禁止为同一概念创建重复令牌。
- 同时设计明暗主题，不把“双色”实现成单一深色界面。

### 4. 实现交互状态

- 覆盖默认、悬停、按下、焦点、禁用、加载、空结果、成功、部分成功、警告、失败和危险确认。
- 主流程下拉选择使用应用内键盘可操作列表框，不回退为 WebView 原生菜单。
- 危险操作与普通操作分离；取消、删除、批量管理和导出保持各自语义。
- 页面不得显示演示业务数据冒充真实数据；后端无数据时展示明确空状态。

### 5. 验证

- 运行受影响组件的测试、类型检查和前端构建。
- 同时检查桌面与窄窗、浅色与深色、键盘焦点与减少动态效果。
- 本地桌面 UI 的最终结论必须来自本次打包产物；核对目标 `.app` 或可执行文件绝对路径与 PID，避免误验旧实例。
- 窗口外框截图必须包含桌面背景，并在像素级检查四角透明、圆角和黑色矩形残边。
- 用户明确表示缺陷仍存在时，停止猜测性补丁，先复现其当前实例并与正确基准做 A/B 对比。

## 交付

- 简要说明界面结果、设计系统如何影响修改以及验证范围。
- 列出可能破坏的内容，并建议覆盖它们的测试。
- UI 修改包验证后只提交本包文件；除非用户明确要求，否则不推送。
