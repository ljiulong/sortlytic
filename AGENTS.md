# 项目级规则

- 本地桌面应用的实机 UI 测试必须以打包后的应用产物为准；不得只用浏览器开发服务器页面代替最终应用测试。
- 每个独立修改包完成并验证后必须立即提交，只提交该包实际修改的文件；不得把多个已完成包长期堆在工作区，也不得顺带提交其他包或既有改动。除非用户明确要求，否则不要推送。
- 前端重设计中用户要求“双色”时，默认实现可切换并持久化的明暗双主题，不得只提供深色方案；如果“双色”可能指两种品牌色，再先澄清具体含义。
- 并行代理因 429、临时限流或服务端重试耗尽失败时，主代理应自动重试一次；再次失败后立即由主线程接管，不得让临时代理故障中断、缩小或重定义用户目标。
- 对于签名、发布、推送、依赖或环境配置等可由 Agent 自行排除的阻塞点，必须直接调查并处理，不等待用户授权或人工介入；仅在需要新增外部权限且无法通过现有凭据解决时报告阻塞。
- 所有代码或配置改动必须先建立可见的执行计划，按独立修改包逐项推进，并在每个包完成后更新计划状态、验证并提交。
- Sortlytic 是本应用正式英文名，不得写成候选序号来源、临时代号或“产品第五个英文名”。
