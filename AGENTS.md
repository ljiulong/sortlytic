# 项目级规则

- `plan/` 是本地计划目录，必须加入 Git 忽略规则，不纳入 Git 提交或 GitHub 发布；不得删除本地 `plan/` 文件，也不得在本修改包中执行 `git rm --cached`，已跟踪内容由主线程分批处理。
- 本地桌面应用的实机 UI 测试必须以打包后的应用产物为准；不得只用浏览器开发服务器页面代替最终应用测试。
- 每个独立修改包完成并验证后必须立即提交，只提交该包实际修改的文件；不得把多个已完成包长期堆在工作区，也不得顺带提交其他包或既有改动。除非用户明确要求，否则不要推送。
- Git 提交信息默认使用英文 Conventional Commits 格式；不要因为中文交流规则或仓库中已有中文提交而改用中文，除非用户明确要求中文提交信息。
- 前端重设计中用户要求“双色”时，默认实现可切换并持久化的明暗双主题，不得只提供深色方案；如果“双色”可能指两种品牌色，再先澄清具体含义。
- 并行代理因 429、临时限流或服务端重试耗尽失败时，主代理应自动重试一次；再次失败后立即由主线程接管，不得让临时代理故障中断、缩小或重定义用户目标。
- 对于可独立推进且不会争用同一热点文件或实机环境的长任务，默认使用子代理并行执行；主线程负责集成、冲突处理和必须串行的打包实机验收，不得在已具备安全并行条件时无故全部串行。
- 对于签名、发布、推送、依赖或环境配置等可由 Agent 自行排除的阻塞点，必须直接调查并处理，不等待用户授权或人工介入；仅在需要新增外部权限且无法通过现有凭据解决时报告阻塞。
- 所有代码或配置改动必须先建立可见的执行计划，按独立修改包逐项推进，并在每个包完成后更新计划状态、验证并提交。
- Sortlytic 是本应用正式英文名，不得写成候选序号来源、临时代号或“产品第五个英文名”。
- README、官网和发布物展示的 Logo 默认必须复用应用实际打包使用的图标资源；不得根据应用内局部图标另行绘制不同品牌标识，除非用户明确要求设计新 Logo。
- 用户提供 Logo 参考图时，必须先核对图形语义和实际资源，禁止用近似勾形的临时 SVG 冒充应用 Logo；没有可复用资源时要明确按参考图复刻并用打包应用验证。
- 用户要求应用窗口圆角时，必须以打包后的 macOS 应用外框为验收对象，分别确认原生窗口四角与前端内容容器，不能只凭页面 CSS 圆角判断完成。
- macOS 无装饰圆角窗口验收时，必须同时检查四角外侧是否残留黑色阴影或矩形边框；只要外缘仍是黑色直角，就不能视为圆角完成。
- macOS 窗口外框验收截图必须包含窗口外侧的桌面背景；只截取窗口内容或被工具裁切到窗口边界的截图，不能作为外缘黑边、阴影或直角已经消失的证据。
- macOS 应用打包和实机验收前必须先检查是否存在 dev app、旧版应用或同 bundle identifier 进程；必须结束旧实例，启动本次新产物，并核对验收进程 PID 对应的可执行文件绝对路径，禁止把重新激活旧进程当作新包已启动。
- macOS 实机自动化不得只按应用显示名或 bundle identifier 定位 Sortlytic；每次界面操作前必须绑定并复核本次目标产物的 `.app` 或可执行文件绝对路径。若自动化工具意外唤醒其他安装位置的同名实例，必须立即停止操作、结束误启实例并重新核对唯一 PID，禁止让新旧实例同时访问同一工作区。
- 用户确认某一版 macOS 窗口圆角正确时，该版本必须作为圆角验收基准；修复外框黑边不得顺带改动圆角半径、透明度或原生圆角实现，并且用户指出 dev app 问题时必须启动、核对并验收 `target/debug` 对应进程，不能用 release 包替代。
- 用户明确指出同一可见缺陷仍存在时，用户现场结论优先于自动测试和既有截图；必须立即停止提交同类猜测性补丁，复现用户当前运行实例并与首个正确版本做 A/B 对比，取得能定位具体绘制层的证据后才能继续修改。
- macOS 圆角窗口的四角必须放大到像素级验收；黑色、白色或任何颜色的不透明矩形底层都属于外壳残边，原生 `cornerRadius`、测试通过或远景截图均不能替代四角直接透出桌面的视觉结果。
- 对于用户已确认来源可信的 macOS 未签名或未公证应用，处理“应用已损坏，无法打开”时，应提供仅针对该应用移除 `com.apple.quarantine` 隔离属性的 `xattr` 命令并说明风险边界；不得只建议等待公证版本，也不得默认指导用户全局关闭 Gatekeeper。
- 采集计划提供年龄范围筛选时，必须将它实现为可选的单一闭区间；启用后只接受接口或公开资料明确返回且位于区间内的年龄，未知、异常或推断年龄不得进入结果。
- 采集性别筛选必须为可选条件；启用后只能使用接口或公开资料明确返回的性别，禁止根据头像、姓名、简介等推断，未知、异常或不匹配账号不得进入结果，并且筛选必须在账号合并后执行。
- 表单式采集的新建态不得预填任何具体任务实例；平台、数据类型、国家地区、采集目标、时间范围、记录数与预算必须由用户为本次任务明确填写，示例只能作为不参与提交的占位提示。
- 正式应用首页只能展示真实工作区数据或明确空状态，禁止回退到演示/预览业务数据；“新建任务”与“任务”必须使用独立页面，导出操作下沉到具体任务并由用户单独选择 Excel 或 PDF，不设置强制同时生成两种格式的全局导出门禁。
- 采集任务的首版原始结果明细只通过 Excel 工作簿导出；PDF 必须是基于真实落库数据和可追溯证据生成的数据分析报告，包含分析结论、样本概况、证据来源与限制说明，不得逐行复制原始账号明细或用占位文案伪装分析完成。没有可用分析结果时不得把 PDF 标记为导出成功。
- 独立页面的主标题必须反映当前页面，工作区名称只能作为次要上下文；没有检查栏的页面不得复用预留空列的双栏网格，任务编辑、取消、运行和导出不得挤在固定窄列中，必须使用清晰分区的横向操作栏并以打包应用实机截图验收。
- 页面顶栏的上下文标签必须随当前页面变化，禁止在首页、新建任务、任务、设置和使用指南中重复显示同一个工作区名称作为统一抬头。
- 浅色主题中的并列事实指标不得为了突出主项而单独使用反相纯黑背景，避免误读为选中态、告警态或深色主题残片；没有明确业务语义时，层级只能通过字号、字重、留白、浅色表面和细分隔建立。
- 同一事实网格的所有相邻项目必须使用统一且可见的细分隔体系，不能因拆分为不同容器而丢失第一条边界；桌面、窄窗与明暗主题都必须实机检查分隔连续性。
- 主流程的平台和国家地区选择不得回退为浏览器或 WebView 原生下拉菜单；必须使用应用内统一、可键盘操作、可见焦点明确且窄窗不溢出的列表框。
- 国家地区选择必须保留全部 249 个唯一 ISO 两位代码，每项同时显示中文名、英文名和两位代码，并支持按中文名、英文名或代码搜索；提交值只能使用标准两位代码。
- 使用指南必须覆盖本地工作区边界、数据源与模型配置、计划校验、确认运行、任务管理、逐任务导出、常见故障、安全和成本边界，不能退化为只含少量摘要步骤的装饰页面。
- 取消任务与删除任务必须是两个独立动作：取消只停止执行并保留记录，删除需二次确认并清理关联本地数据；成功、部分成功、失败和已取消等终态任务也必须提供删除入口。
- API 配置测试必须向当前配置的真实服务端点发出最小安全请求并验证可用响应；字段完整性只能作为联网前置检查，不能单独标记为“测试成功”或允许激活配置。
- TikHub 配置测试成功后必须展示充值余额、免费额度、可用总额、当日用量和最近检测时间；缺失或读取失败的金额必须明确显示不可用，禁止伪装为零或沿用历史成功值。
- 自然语言采集必须提供可查看、可新建版本和显式激活的 AI 提示词设置；当前激活提示词正文、版本、Schema 与证据约束必须真实进入模型请求和运行快照，提示词激活回归也必须调用当前已测试的真实 AI 配置，禁止用本地关键词规则或固定 JSON 冒充模型调用。
- 多端点实时计价的限流验收不得只验证最大并发数为 1；必须覆盖串行请求时间窗口、HTTP/业务码 429 的 `Retry-After` 或有限退避、连续及并发预检复用、部分成功报价续用，并在打包应用的真实多端点计划中取得非 429 的预算或额度业务结论后才能交付。
- 额度上限、余额上限或非阻塞启动逻辑变更时，必须按 0.1 美元递增覆盖 0.1–1.0 美元每一档，每档至少重复 3 轮；验收必须分别证明计划可启动、精确到达设定上限时不超支、下一次请求将超过设定上限或实时可用余额时在发出采集请求前自动停止。
- 任务确认运行成功后必须自动进入该任务的独立预览；导出成功必须在当前任务上下文展示文件绝对路径并提供可用的打开动作，失败必须显示明确错误；任务页必须支持选择全部可删除任务和批量删除，使用统一二次确认，已排队与运行中任务不得进入批量删除集合。
- 任务成功状态必须基于经过契约解析并完成落库的真实响应证据；同一 API 响应同时包含兼容空数组与主要非空记录数组时，必须优先解析 endpoint 的主要记录字段，禁止把实际有结果的响应记为“成功、0 条”。真实响应确实为空时可以正常完成，但全流程验收必须明确区分“无匹配结果”和“解析后丢失结果”。
- 任务页新增卡片、批量工具栏、选择控件、确认栏或导出回执不得使用裸表单控件、调试式路径框或重复嵌套的同色灰色卡片直接交付；必须沿用现有任务卡的排版、间距、按钮高度和表面层级，选中态需有整卡反馈，导出绝对路径与打开动作需组成紧凑状态带，并以打包后的 macOS 应用同时验收桌面与窄窗、明暗主题。
- 成功或部分成功任务必须提供应用内结果查看入口，进入任务后展示当前任务最新运行已落库的真实结果数据；不得只显示结果数量或要求用户导出后查看。结果视图必须区分加载、空结果和读取失败状态，并保留逐任务导出能力。
- 任务删除选择框不得在任务卡默认态常驻；只有用户显式进入批量管理模式后才显示任务选择框、全选、已选数量和批量删除动作，退出批量管理时必须清空选择并恢复干净的任务卡默认态。
- 结果数据的缺失值必须区分计划配置与采集结果：用户未启用年龄、性别等可选条件时显示“任务未设置”，已要求或默认采集但接口未返回时显示“未采集到”；后端明确返回的数值 `0` 必须保留为 `0`，禁止与缺失值混淆或统一显示为“未提供”。
- 可以通过本地仓库、文件、日志、数据库、测试结果或现有产物可靠判断的问题，直接依据本地数据回答，不需要联网核实；只有本地信息不足、需要查证外部现状或用户明确要求联网时才使用联网搜索。
