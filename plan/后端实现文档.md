# 后端实现文档

版本：v0.2
日期：2026-07-17
关联文档：`智能数据整理平台_PRD.md`、`技术框架总览.md`、`后端技术方案.md`、`数据库与本地存储方案.md`、`前端技术方案.md`
适用范围：macOS 单端 MVP
文档定位：定义 Tauri v2 本地后端的实现边界、模块拆分、命令接口、存储职责、任务队列、外部请求、AI 运行、导出、安全和测试策略

> 决策覆盖：从 v0.2 起，用户确认的统一 JSON API 配置方案覆盖本文及旧文档中“API Key 必须进入 macOS Keychain／系统安全存储”的既有口径。TikHub 与 AI API 配置以当前工作区 `secrets/api-config.json` 为唯一主数据源；旧 Keychain 条目不由新版本读取或删除。当前 AI API 只做保存、完整性校验、查看和切换，自然语言计划仍由本地规则引擎生成。

## 1. 文档目标与范围

### 1.1 文档目标

本文档用于指导 macOS MVP 的本地后端实现。这里的“后端”不是云端服务器，而是运行在用户本机的 Tauri Rust 后端、领域服务层、本地数据库、本地文件系统、任务队列和外部请求适配层。

实现目标：

- 通过 Tauri v2 commands 向前端暴露受控能力。
- 在 Rust 后端统一承接工作区、密钥、数据库、任务队列、TikHub 请求、模型供应商请求、AI 运行、导出、Webhook、备份和安全边界。
- 保证所有任务、数据、AI 输出、报告和导出文件都能在本地追溯。
- 保证 TikHub API Key 与 AI API Key 的明文只进入当前工作区私有 `secrets/api-config.json`，不进入 SQLite、日志、审计、导出、Webhook、备份或 AI prompt。
- 保证自然语言采集由本地规则引擎生成结构化采集计划，不直接调用 TikHub，不读取密钥。
- 保证 MVP 内置提示词优化、提示词版本管理、Schema 约束、证据引用和基础回归样例。
- 保证对外导出以 Excel 工作簿和 PDF 报告为主。

### 1.2 MVP 范围

MVP 只覆盖：

- macOS 单端本地工作区。
- 单用户本地任务。
- TikTok、抖音、小红书三平台公开数据采集。
- TikHub REST API 作为采集通道。
- OpenAI、Anthropic、Gemini、Ollama、自定义 OpenAI-compatible endpoint 等 AI API 配置的保存、完整性校验、查看与切换；当前不调用真实模型。
- 本地 SQLite、本地文件系统、工作区私有 API 配置 JSON。
- Excel 工作簿、PDF 报告、本机 Webhook 摘要发送。

MVP 不做：

- 不做账号体系。
- 不接远端数据库。
- 不做远端同步或多设备自动同步。
- 不做云端任务。
- 不做云端密钥托管。
- 不做团队协作后端。
- 不做自研爬虫。
- 不采集非公开数据。
- 不做自动评论、自动私信、自动发布等平台交互动作。
- 不把 TikHub MCP 作为核心执行路径。

### 1.3 Tauri v2 实现基线

后端实现必须遵守 Tauri v2 的能力边界：

- 使用 `tauri::command` 暴露 Rust 后端入口。
- 使用 `tauri::Builder::manage` 注册应用级状态。
- 在 command 中通过 `tauri::State` 读取应用状态，例如工作区上下文、数据库池、任务队列、API 配置注册表、请求客户端。
- 文件访问使用 `tauri-plugin-fs`，并在 capabilities 中配置最小权限范围。
- WebView 加载本地文件时使用 asset protocol，并配置 `app.security.assetProtocol.scope`，只允许工作区导出、报告预览和必要缓存目录。
- 前端不得绕过 commands 直接访问密钥、SQLite、TikHub、模型供应商或任意本地文件路径。

## 2. 后端分层架构

### 2.1 总体分层

```text
前端应用层
  - React 页面、表单、表格、报告预览、任务状态展示
  |
  v
Tauri Command 层
  - 受控命令入口、入参反序列化、统一错误返回、权限和状态检查
  |
  v
应用服务层
  - 工作区、密钥、供应商、任务、采集、AI、数据资产、导出、Webhook、备份
  |
  v
领域服务层
  - TikHub 适配、模型适配、提示词、Schema、证据、报告模型、成本、状态机
  |
  v
基础设施层
  - SQLite、文件系统、工作区私有 API 配置注册表、HTTP 客户端、任务队列、日志、迁移
  |
  v
外部服务
  - TikHub REST API、模型供应商 API、用户配置的 Webhook 目标
```

### 2.2 Command 层职责

Command 层只负责对前端暴露明确、可审计、可序列化的接口，不承载复杂业务逻辑。

职责：

- 定义 `#[tauri::command]` 函数。
- 读取 `tauri::State<AppState>`。
- 校验入参基础格式。
- 调用应用服务层。
- 把内部错误转换为统一 `AppError`。
- 对敏感返回值做脱敏。

禁止：

- 在 command 中直接拼接 TikHub 或模型供应商请求。
- 在 command 中直接读写明文密钥。
- 在 command 中直接写复杂 SQL。
- 在 command 中把原始外部错误堆栈返回给前端。

### 2.3 应用级状态

应用启动时通过 `tauri::Builder::manage` 注册本地后端共享状态。

建议状态：

```text
AppState
  workspace_registry
  active_workspace
  database_pool
  job_queue
  provider_registry
  tikhub_registry
  http_client
  api_profile_registry
  file_store
  event_bus
  shutdown_signal
```

说明：

- `workspace_registry` 管理最近打开的本地工作区列表。
- `active_workspace` 表示当前工作区路径、目录结构和 Schema 版本。
- `database_pool` 管理 SQLite 连接池或连接句柄。
- `job_queue` 管理本地任务调度、取消和失败重试。
- `provider_registry` 是未来模型执行使用的派生能力视图，不得覆盖 `api-config.json` 中的配置主数据。
- `tikhub_registry` 是 TikHub 平台、数据类型、接口白名单和成本估算规则的单一事实源。
- `api_profile_registry` 封装 `secrets/api-config.json` 的版本校验、私有权限校验、原子读写、并发锁与安全视图。
- `file_store` 封装工作区目录、导出目录、临时目录和文件哈希。
- `event_bus` 向前端推送任务状态、日志、导出进度等事件。

### 2.4 应用服务层

应用服务层面向业务用例组织代码：

- `WorkspaceService`：工作区初始化、打开、健康检查、存储占用、备份恢复。
- `ApiProfileService`：TikHub 与 AI 命名配置的保存、测试或校验、激活、删除、凭据解析和 SQLite 镜像同步。
- `TaskService`：任务创建、确认、排队、取消、失败重试、复制、日志查询。
- `CollectionService`：采集计划校验、成本估算、TikHub 调用、原始响应保存、标准化。
- `AiService`：未来真实模型执行的 AI 整理、运行快照、Schema 校验和证据保存；当前不读取 AI 当前配置，不发模型请求。
- `PromptService`：提示词模板、版本、回归样例、回滚和变更说明。
- `DataAssetService`：原始数据、标准化数据、AI 结果、证据、报告索引查询。
- `ExportService`：导出前检查、Excel 生成、PDF 生成、文件落盘、导出记录。
- `WebhookService`：Webhook 配置、测试、发送摘要、日志记录。
- `BackupService`：手动备份、备份 manifest、恢复前检查。

### 2.5 领域服务层

领域服务层沉淀跨入口复用的规则：

- TikHub 平台能力注册。
- TikHub 参数白名单。
- 国家/地区标准化代码、来源、置信度和校验状态。
- 模型协议适配。
- Prompt Runtime Snapshot。
- JSON Schema 校验。
- 字段级证据生成。
- 成本估算与成本上限。
- 导出完整性门禁。
- 日志脱敏与审计规则。

### 3.2 模块拆分原则

- `commands/` 只放 Tauri command 入口。
- `services/` 放用例编排，不直接保存明文密钥。
- `domain/` 放状态机、实体、值对象、错误类型和业务规则。
- `infrastructure/` 放 SQLite、文件系统、安全存储、HTTP、日志、事件等基础设施。
- `adapters/tikhub/` 放 TikHub REST API 请求构造、响应解析和平台标准化。
- `adapters/models/` 放不同模型协议的适配，不把供应商硬编码进 AI Service。
- `jobs/` 放本地任务队列、执行器、取消和重试逻辑。
- `exporters/` 放 Excel、PDF、报告模型和导出完整性检查。
- `security/` 放脱敏、审计、权限范围和敏感字段检查。

## 4. Tauri command 接口清单

### 4.1 统一约定

所有 command 使用统一返回结构：

```text
Result<T, AppError>
```

`AppError` 字段：

```text
code
message
stage
retryable
safe_details
```

规则：

- `message` 面向用户或前端展示，应明确可执行。
- `safe_details` 必须脱敏。
- 不返回完整 API Key、Authorization Header、Cookie、完整请求 Header、原始错误堆栈。
- command 命名使用动词加对象，保持前端 invoke 清晰。

### 4.2 工作区 commands

| 命令 | 用途 | 关键入参 | 返回 | 边界 |
| --- | --- | --- | --- | --- |
| `create_workspace` | 创建本地工作区 | 名称、根目录 | 工作区摘要 | 创建目录、SQLite、基础表和默认模板 |
| `open_workspace` | 打开已有工作区 | 工作区路径 | 工作区摘要 | 校验 Schema 版本和目录结构 |
| `get_active_workspace` | 获取当前工作区 | 无 | 工作区摘要 | 不返回敏感路径外的文件列表 |
| `rename_workspace` | 修改工作区名称 | 工作区 ID、名称 | 工作区摘要 | 只改 SQLite 元数据 |
| `get_workspace_storage_usage` | 统计本地占用 | 工作区 ID | 占用摘要 | 按数据库、原始响应、导出、日志、备份分组 |
| `run_workspace_health_check` | 本地健康检查 | 工作区 ID | 检查结果 | 执行数据库 quick check、路径可写、缺失文件扫描 |
| `close_workspace` | 关闭当前工作区 | 工作区 ID | 成功状态 | 等待任务安全停止或提示仍有任务运行 |

### 4.3 API 配置注册表 commands

| 命令 | 用途 | 关键入参 | 返回 | 边界 |
| --- | --- | --- | --- | --- |
| `get_api_profile_registry` | 获取配置注册表 | 无 | 两类安全配置视图与当前 ID | 只返回脱敏密钥，不返回或复制完整 API Key |
| `save_api_profile` | 新建或编辑配置 | `kind`、配置字段、可选密钥 | 安全配置视图 | 编辑时密钥留空表示保留；保存后同步 SQLite 镜像 |
| `test_api_profile` | 测试或校验配置 | `kind`、配置 ID | 安全测试结果 | TikHub 真实测试；AI 只做本地完整性校验 |
| `activate_api_profile` | 设置当前配置 | `kind`、配置 ID | 注册表安全视图 | 只允许已通过测试或校验的配置 |
| `delete_api_profile` | 删除配置和凭据 | `kind`、配置 ID | 注册表安全视图 | 二次确认由前端发起；写脱敏审计 |

配置规则：

- TikHub 与 AI 均支持多条命名配置，每类最多一条当前配置；同一 AI 供应商允许多个账号或端点。
- 第一条有效配置自动激活，后续新增配置不抢占当前配置。删除当前配置后指针置空，不静默选择其他配置。
- 被运行中或恢复中不可变快照引用的 TikHub 配置不得编辑或删除，但允许切换当前配置。
- SQLite 只保存可从 JSON 重建的配置镜像与 `secret_ref` 元数据，不保存明文密钥，也不得反向覆盖 JSON。
- 前端、command 返回、审计、日志、导出、Webhook 与备份均不得包含完整 API Key。

### 4.4 兼容入口边界

旧密钥、TikHub 单例和模型供应商 CRUD 只允许在前端迁移期间作为内部兼容层存在。前端迁移完成后必须从 Tauri invoke handler 移除外部变更入口，防止绕过 JSON 主数据源；后端运行镜像和内部读取函数可以保留。

### 4.5 任务 commands

| 命令 | 用途 | 关键入参 | 返回 | 边界 |
| --- | --- | --- | --- | --- |
| `create_collection_task` | 创建采集任务草稿 | 来源、名称、平台、数据类型 | 任务视图 | 不发起外部请求 |
| `update_collection_task` | 更新任务草稿 | 任务 ID、字段 | 任务视图 | 只允许草稿或等待确认状态 |
| `estimate_task_cost` | 估算成本 | 任务 ID 或采集计划 | 成本估算 | 不产生正式采集费用 |
| `confirm_collection_plan` | 确认采集计划 | 任务 ID、计划 ID | 任务视图 | 通过 Schema、白名单和成本检查后进入可排队状态 |
| `enqueue_task` | 加入任务队列 | 任务 ID | 任务运行视图 | 不冻结 TikHub 配置；首次领取时才绑定当前配置 |
| `cancel_task` | 取消任务 | 任务 ID | 任务视图 | 取消后不得发起新的外部请求 |
| `retry_task` | 重试失败任务 | 任务 ID、阶段可选 | 新任务运行视图 | 仅重试可重试阶段 |
| `copy_task` | 复制任务 | 任务 ID | 新任务草稿 | 不复制运行结果 |
| `get_task` | 获取任务详情 | 任务 ID | 任务视图 | 包含状态和成本摘要 |
| `list_tasks` | 查询任务列表 | 筛选、分页 | 任务分页 | 支持平台、状态、来源、时间筛选 |
| `list_task_logs` | 查询任务日志 | 任务运行 ID、分页 | 日志分页 | 只返回脱敏日志 |

### 4.6 采集 commands

| 命令 | 用途 | 关键入参 | 返回 | 边界 |
| --- | --- | --- | --- | --- |
| `list_supported_platforms` | 列出平台能力 | 无 | 平台能力列表 | 只返回 TikTok、抖音、小红书 |
| `list_platform_data_types` | 列出数据类型 | 平台 | 数据类型列表 | 包含是否支持国家/地区筛选 |
| `validate_collection_params` | 校验表单参数 | 平台、数据类型、参数 | 校验结果 | 只做本地 Schema 和白名单校验 |
| `generate_form_collection_plan` | 表单生成采集计划 | 表单参数 | 采集计划 | 不发起正式 TikHub 请求 |
| `preview_collection_plan` | 预览计划和成本 | 采集计划 | 预览摘要 | 不产生正式采集费用 |

TikHub 正式采集不通过单独 command 直接调用，而由任务队列在用户确认并入队后执行。

### 4.7 自然语言计划与未来 AI commands

当前 `generate_collection_plan_from_text` 使用本地规则引擎，不读取 AI 当前配置。其余真实模型整理 commands 在未来模型执行链路接入后启用。

| 命令 | 用途 | 关键入参 | 返回 | 边界 |
| --- | --- | --- | --- | --- |
| `generate_collection_plan_from_text` | 自然语言生成采集计划 | 任务 ID、自然语言 | 采集计划 | 本地规则引擎只产出计划，不调用 TikHub 或 AI API |
| `validate_ai_collection_plan` | 校验结构化计划 | 计划 ID | 校验结果 | Schema、白名单、成本、缺失条件检查 |
| `run_ai_analysis` | 未来创建 AI 整理任务 | 任务 ID、模板、模型选择 | 任务运行视图 | 当前不启用；未来由队列执行 |
| `get_ai_run` | 获取 AI 运行记录 | AI 运行 ID | AI 运行视图 | 可查看 Runtime Snapshot |
| `list_ai_runs` | 查询 AI 运行 | 任务 ID、类型 | AI 运行列表 | 不返回明文密钥 |
| `list_field_evidence` | 查询字段级证据 | 任务 ID、字段可选 | 证据列表 | 支持从洞察跳回原始记录 |

## 5. SQLite 与文件系统职责边界

### 5.1 SQLite 职责

SQLite 存储结构化索引、状态、关系和可追溯元数据。

必须进入 SQLite：

- 工作区元数据。
- 密钥引用 `secret_ref`，不含明文。
- 模型供应商配置。
- 模型能力和成本策略。
- 提示词模板、提示词版本、回归样例、回归运行结果。
- 采集任务、自然语言需求、采集计划、API 调用步骤。
- 任务运行、任务日志、成本估算和实际成本。
- 原始记录索引、文件路径、哈希、摘要。
- 标准化记录。
- AI 运行记录和 Runtime Snapshot。
- 字段级证据。
- 洞察、报告数据模型、导出任务、Webhook 记录、审计日志。

SQLite 初始化建议：

```sql
PRAGMA foreign_keys = ON;
PRAGMA journal_mode = WAL;
PRAGMA wal_autocheckpoint = 1000;
PRAGMA synchronous = NORMAL;
PRAGMA temp_store = MEMORY;
```

维护命令：

```sql
PRAGMA optimize;
PRAGMA quick_check;
PRAGMA wal_checkpoint(TRUNCATE);
```

### 5.2 文件系统职责

文件系统保存大体积、可导出、可预览或临时中间文件。

建议目录：

```text
workspace/
  app.sqlite
  app.sqlite-wal
  app.sqlite-shm
  raw/
    tikhub/
      {task_id}/
  exports/
    excel/
    pdf/
  reports/
    {report_id}/
  prompts/
    snapshots/
  logs/
  temp/
  backups/
```

文件系统保存：

- TikHub 原始响应文件。
- 模型原始输出文件。
- Excel 工作簿。
- PDF 报告。
- 报告模型快照。
- 提示词快照。
- 脱敏日志。
- 备份包。
- 临时文件。

规则：

- SQLite 保存文件路径、哈希、大小、摘要和生成时间。
- 原始响应不直接塞入 SQLite。
- 导出成功后记录文件哈希和大小。
- 临时文件可清理，用户保存的任务、报告、导出索引、提示词版本、AI 运行记录和字段级证据不可自动清理。

### 5.3 工作区私有 API 配置注册表职责

当前工作区 `secrets/api-config.json` 是 TikHub 与 AI API 配置的唯一主数据源，统一保存多条配置、每类当前配置 ID 和与配置对象分离的明文凭据。

规则：

- `secrets/` 固定为 `0700`，`api-config.json` 固定为 `0600`；拒绝符号链接、非普通文件、宽松权限和无法可靠落实私有权限的文件系统。
- 写入使用同目录临时文件、文件同步和原子重命名，进程内读写由同一互斥锁串行化。
- JSON 损坏、Schema 版本未知或路径不安全时失败关闭，不覆盖原文件；API 配置管理和新任务领取停止，已有工作区与历史数据仍可浏览。
- API Key 以明文保存在 JSON 的 `credentials` 对象中。私有权限只能限制其他本机用户，不能防止同一 macOS 账户下的恶意进程读取。
- 旧 SQLite 只导入非敏感配置元数据，配置统一标记为 `needs_rebind`，当前指针置空；迁移不读取或删除旧 Keychain 条目。
- `api-config.json` 不进入应用备份、Excel、PDF、日志、Webhook 或前端文件访问范围。
- 当前统一注册表不接受 Webhook 敏感 Header；在独立私有存储方案实现前，不得持久化这类 Header。

### 5.4 Tauri 文件权限范围

文件访问必须遵守最小权限：

- `tauri-plugin-fs` 只允许访问当前工作区目录、用户选择的导出目标目录、应用资源目录。
- 不允许前端任意读取 `$HOME`。
- 不允许前端直接读取密钥、数据库文件或日志原文。
- 所有敏感文件读写由 Rust 后端封装。

### 5.5 WebView 本地文件预览

WebView 预览本地文件时必须配置 asset protocol。

要求：

- 在 `app.security.assetProtocol` 中启用 asset protocol。
- `scope` 只包含报告预览、导出预览和必要缓存目录。
- 不把整个用户主目录加入 scope。
- PDF、图片、报告预览素材必须先通过后端确认属于当前工作区允许范围。
- 如果文件不在 scope 内，前端不得通过路径绕过加载。

## 6. 任务队列与状态机

### 6.1 任务状态

任务主状态：

```text
draft
waiting_confirmation
queued
running
partial_success
success
failed
cancelled
needs_review
```

中文含义：

- `draft`：草稿。
- `waiting_confirmation`：等待用户确认。
- `queued`：排队中。
- `running`：运行中。
- `partial_success`：部分成功。
- `success`：成功。
- `failed`：失败。
- `cancelled`：已取消。
- `needs_review`：待人工确认。

### 6.2 任务阶段

标准阶段：

```text
参数校验
成本估算
用户确认
TikHub 采集
原始响应落盘
标准化
AI 整理
Schema 校验
证据生成
报告模型生成
导出
Webhook
```

不是所有任务都必须经过全部阶段。例如只采集不导出时可停在标准化阶段；只导出已有报告时从报告模型阶段开始。

### 6.3 状态转换

```text
draft
  -> waiting_confirmation
  -> queued
  -> running
  -> success

running
  -> partial_success
  -> success

running
  -> needs_review

running
  -> failed

queued 或 running
  -> cancelled

failed
  -> queued
```

转换规则：

- 自然语言任务必须先保存 `task_intent` 和 `collection_plan`。
- 本地规则引擎生成的采集计划不得直接进入 `queued`，必须先通过 Schema、白名单、成本检查和用户确认。
- 缺少平台、数据类型、国家/地区、时间范围或预算条件时，进入 `waiting_confirmation` 或 `needs_review`，不得猜测执行。
- 取消任务后不得发起新的外部请求。
- 已发出的请求等待完成或超时，并记录取消时间和已完成阶段。
- 部分成功不得汇总成成功，导出时必须显示部分成功范围。

### 6.4 队列执行器

队列执行器职责：

- 控制本地并发上限。
- 按任务阶段保存进度。
- 支持取消和失败重试。
- 在每个阶段写入脱敏日志。
- 在外部请求前检查取消信号、密钥状态、成本上限和磁盘空间。
- 将任务状态变化通过事件发送给前端。

并发建议：

- TikHub 采集并发按平台和接口限流策略控制。
- 未来真实模型调用并发按供应商和用户成本上限控制；当前 AI 配置不进入队列。
- 导出任务避免与大规模采集高并发执行，防止本地磁盘和内存压力过高。

### 6.5 重试策略

可重试：

- 网络超时。
- TikHub 限流。
- 未来模型执行链路中的模型供应商临时不可用。
- Webhook 目标短暂不可达。
- 可恢复的文件写入失败。

不可直接重试：

- API Key 无效。
- 参数白名单失败。
- Schema 不兼容。
- 成本上限触发。
- PDF 字体缺失。
- 用户取消。

重试记录必须包含：

- 原任务 ID。
- 原任务运行 ID。
- 重试阶段。
- 重试次数。
- 上一次错误摘要。
- 是否复用已采集数据。

## 7. TikHub API 适配与合规边界

### 7.1 接入方式

MVP 只通过 TikHub REST API 采集公开数据。TikHub MCP 不进入 MVP 核心执行路径。

采集入口：

- 表单式精确采集：应用根据用户选择的平台、数据类型、国家/地区、关键词、账号、时间范围生成采集计划。
- 自然语言计划采集：本地规则引擎只把自然语言转为结构化采集计划，应用负责校验和执行。

### 7.2 平台与数据类型

MVP 只覆盖：

- TikTok。
- 抖音。
- 小红书。

数据类型：

- 关键词搜索。
- 视频或笔记详情。
- 用户公开信息。
- 评论采集。

三平台通用能力：

- 基于公开评论、公开互动归因字段或公开视频、笔记、直播间关联账号生成候选账号。
- 作品数为 0 的账号允许入库，并标记为“无公开作品样本”。
- 按国家/地区标准化代码标记或筛选账号。
- 国家/地区字段必须展示来源、置信度和校验状态。
- 直播间状态监测仅限 TikHub 支持的公开接口范围，不接直播 WebSocket。
- 开播触发动作只做本机通知、数据快照、报告生成或用户启用的 Webhook 摘要。

### 7.3 参数白名单

每个 TikHub endpoint 必须在本地注册：

```text
endpoint_key
platform
data_type
required_params
optional_params
region_support
max_page_size
max_request_count
cost_policy
rate_limit_policy
response_schema
normalizer
```

规则：

- 本地规则引擎或未来模型生成的参数必须落在 endpoint 白名单内。
- 不支持国家/地区筛选的接口必须明确返回“不支持该筛选”，不能静默忽略。
- 国家/地区是同一业务概念，不得拆成并列字段造成口径冲突；标准化国家代码只是国家/地区的标准化值。
- 用户确认前不得发起会产生正式采集费用的请求。
- 任何高成本、多平台、多数据类型、直播监测或开播触发任务必须二次确认。

### 7.4 合规边界

禁止：

- 绕过登录、验证码、权限限制。
- 采集非公开粉丝关系、私信、隐藏互动列表或登录后可见数据。
- 自动评论、私信、点赞、关注、发布。
- 将 TikHub API Key 交给 AI prompt 或工具链。
- 使用自研爬虫绕过 TikHub 或平台限制。

报告和导出必须包含数据来源声明；只有实际包含模型生成内容时才增加 AI 免责声明。

## 8. 模型供应商与提示词版本

### 8.1 Provider Registry

当前工作区 `secrets/api-config.json` 是 AI 配置的单一事实源。Provider Registry 仅在未来模型执行链路中作为从当前 AI 配置和运行能力生成的派生视图，不得反向写回或覆盖 JSON。

核心字段：

```text
provider_id
display_name
enabled
auth_type
credential_ref_id
base_url
api_format
region
models
default_model
capabilities
cost_policy
rate_limit_policy
health_check
```

设计规则：

- 不把业务逻辑写死为 OpenAI、Gemini、Claude。
- 供应商和协议格式分离。
- 模型唯一标识使用 `provider_id + model_id`。
- 支持自定义 OpenAI-compatible endpoint。
- 支持 Ollama 或本地模型网关。
- 每次 AI 运行必须保存 Runtime Snapshot。
- 当前默认模型变更不得影响历史任务可复现性。

### 8.2 模型适配层

模型适配层输入：

```text
provider_id
model_id
api_format
base_url
secret_ref
prompt_version
input_messages
output_schema
temperature
max_tokens
timeout
```

模型适配层输出：

```text
raw_output
parsed_json
input_tokens
output_tokens
latency_ms
first_token_latency_ms
retry_count
provider_status
error
```

错误分类：

- 网络错误。
- 认证错误。
- 限流错误。
- 供应商不可用。
- 协议不兼容。
- 非 JSON 输出。
- Schema 校验失败。
- 超时。
- 成本上限触发。

### 8.3 提示词模板

MVP 内置提示词模板：

- 自然语言采集计划生成。
- 缺失条件追问。
- 通用摘要。
- 评论情绪分析。
- 主题分类。
- 实体抽取。
- 代表性观点提取。
- 内容爆点拆解。
- 风险与异常提示。
- 平台差异对比。

提示词优化目标：

- 稳定结构化输出。
- 降低歧义。
- 降低无效 Token 成本。
- 保留证据引用。
- 明确失败边界。
- 减少数据 slop。
- 提高跨供应商兼容性。

禁止通过提示词绕过：

- 参数白名单。
- 成本确认。
- 用户确认。
- 密钥边界。
- Schema 校验。
- 证据引用门禁。

### 8.4 提示词版本

`prompt_version` 必须记录：

```text
template_id
version
content
change_note
status
created_at
activated_at
rollback_from_version
content_hash
```

规则：

- 每次 AI 运行必须绑定 `prompt_version_id`。
- 修改提示词必须生成新版本。
- 内置模板不能直接覆盖，只能派生用户版本。
- 未通过基础回归样例的版本不得设为 active。
- 回滚必须生成审计日志。

### 8.5 Runtime Snapshot

每次 AI 运行保存：

```text
task_id
provider_id
model_id
api_format
base_url_type
prompt_version_id
output_schema_id
capabilities_json
config_source
created_at
```

作用：

- 保证历史任务可复现。
- 保证导出报告能展示模型和提示词版本。
- 保证供应商配置变更后旧任务仍可审计。

## 9. Schema 校验、证据引用与回归样例

### 9.1 Schema 校验范围

必须绑定 Schema 的输出：

- 自然语言采集计划。
- 缺失条件追问。
- AI 整理结果。
- 字段级证据。
- 报告数据模型。
- Webhook 摘要。

校验失败处理：

- 保存原始输出到文件。
- `ai_run.schema_valid` 记为 false。
- 记录错误摘要。
- 任务进入 failed 或 needs_review。
- 不得进入 TikHub 正式采集或核心导出。

### 9.2 采集计划 Schema

采集计划必须包含：

```text
platforms
data_types
region
keywords
accounts
time_range
steps
request_limit
cost_estimate
missing_fields
confidence
requires_user_confirmation
```

规则：

- `platforms` 只能包含 TikTok、抖音、小红书。
- `region` 使用国家/地区字段，标准化国家代码只是存储值或筛选值。
- `steps` 必须映射到已注册 endpoint。
- `missing_fields` 非空时不得进入正式采集。
- `requires_user_confirmation` 为 true 时必须停在等待确认。

### 9.3 字段级证据

本节在未来真实模型执行链路接入后启用。模型生成字段必须保存：

```text
target_entity_type
target_entity_id
field_name
generated_value
source_record_ids
source_field_paths
transform_reason
confidence
validation_status
review_status
ai_run_id
prompt_version_id
```

门禁：

- 核心洞察没有来源记录 ID 时，不得进入核心报告。
- 证据缺失、来源记录不存在、字段冲突时进入待人工确认。
- AI 推断的国家/地区、语言、受众市场、情绪、主题、风险、意图必须带置信度和证据引用。
- 待人工确认不得汇总成成功。

### 9.4 基础回归样例

每个核心提示词模板必须有基础回归样例。

采集计划样例至少覆盖：

- 正常自然语言需求。
- 缺少平台。
- 缺少数据类型。
- 缺少国家/地区。
- 缺少时间范围。
- 多平台高成本任务。
- 本地规则引擎或未来模型生成不在白名单内的参数。

AI 整理样例至少覆盖：

- 正常摘要。
- 评论情绪分析。
- 主题分类。
- 实体抽取。
- 证据不足。
- 字段冲突。
- 模型输出非 JSON。
- 输出缺少来源记录 ID。

回归运行记录：

```text
template_id
prompt_version_id
provider_id
model_id
case_id
status
schema_valid
rules_valid
error_summary
raw_output_path
created_at
```

要求：

- 修改提示词后必须运行基础回归样例。
- 失败样例必须保留差异摘要。
- 固定样例应优先使用小输入，避免回归成本过高。

## 10. Excel、PDF、Webhook 导出实现

### 10.1 导出总原则

导出前必须执行完整性检查：

- 报告数据模型存在。
- 核心洞察有证据引用。
- 待人工确认字段已标记。
- 部分成功范围已标记。
- 失败步骤已标记。
- Excel 工作表完整。
- PDF 字体可用。
- 不包含 API Key。
- 不包含完整请求 Header。

### 10.2 Excel 工作簿

MVP 对外数据导出默认使用 Excel 工作簿。

工作表：

- `任务概览`：任务名称、平台、数据类型、国家/地区、时间范围、状态、成本、模型。
- `原始数据摘要`：原始记录 ID、平台、原始链接、采集时间、原始文件哈希、摘要。
- `标准化数据`：作者、正文、链接、发布时间、互动指标、标签、国家/地区、来源、置信度。
- `AI结构化结果`：摘要、情绪、主题、实体、风险、置信度、校验状态。
- `洞察与证据`：洞察、证据记录 ID、原始链接、模型、提示词版本、人工确认状态。
- `运行日志`：阶段、时间、状态、错误摘要。
- `成本明细`：TikHub 请求量、模型 Token、模型成本、有效入库成本、过滤损耗。

实现要求：

- 支持中文。
- 表头冻结。
- 关键列宽可读。
- 长链接可点击或可复制。
- 大任务使用分批写入或流式写入，避免 UI 卡死。
- 单元格不得出现明文 API Key。

### 10.3 PDF 报告

PDF 是报告交付格式，和 Excel 共用同一份报告数据模型。

报告结构：

- 报告标题。
- 生成时间。
- 任务条件。
- 平台范围。
- 数据概览。
- 成本概览。
- 核心洞察。
- 主题分类。
- 情绪分析。
- 代表性记录。
- 证据引用。
- 模型与提示词版本。
- 免责声明。

实现要求：

- 使用可商用中文字体，例如思源黑体或 Noto Sans SC。
- 字体缺失时阻止生成 PDF，并提示用户修复。
- 支持中英文混排。
- 支持链接。
- 支持基础表格。
- 支持分页、页眉页脚和报告元数据。
- PDF 组件不直接读取数据库，只接收报告数据模型。
- 生成后校验文件大小、页数、标题和路径。

### 10.4 Webhook 摘要

Webhook 是 n8n 等自动化工具的轻集成路径，不是核心卖点。

发送内容：

```text
task_id
task_name
platforms
status
record_count
core_summary
export_file_paths
generated_at
```

禁止发送：

- API Key。
- 完整 Header。
- Cookie。
- 超大原始 JSON。
- 未脱敏错误日志。

规则：

- Webhook 只在本机任务完成时触发。
- 不保证离线补发。
- 成功和失败都要写 `webhook_job`。
- 当前不持久化 Webhook 敏感 Header；未来引入独立私有存储前，Webhook 只允许无敏感 Header 的受控配置。

## 12. 安全与隐私

### 12.1 密钥安全

- TikHub 与 AI API Key 以明文保存在当前工作区私有 `secrets/api-config.json`，目录和文件权限分别为 `0700`、`0600`。
- `0700/0600` 不能防止同一 macOS 账户下的恶意进程读取 JSON，这是取消 Keychain 后明确接受的安全降级。
- SQLite 不保存明文密钥。
- 前端不接收明文密钥。
- AI prompt 不包含 TikHub API Key 或模型供应商 API Key。
- 日志、导出、Webhook、备份不包含完整 API Key。
- 旧 Keychain 条目不自动读取或删除；旧配置必须重新输入一次凭据并通过测试或校验后才能激活。
- 删除当前配置后当前指针置空；新鲜排队任务等待用户显式选择新的当前 TikHub 配置。

### 12.2 网络安全

- 外部请求默认使用 HTTPS。
- 用户自定义本地模型网关可允许本机地址，但必须在配置中明确展示并由用户确认。
- TikHub 请求走统一请求层。
- 模型供应商请求走模型适配层。
- Webhook 发送前展示目标 URL 和摘要范围。

### 12.3 数据隐私

MVP 默认不上传以下数据到自有服务器：

- API Key。
- 原始社交数据。
- 标准化数据。
- AI 输出。
- 报告文件。
- 任务日志。

会发送给外部服务的数据：

- TikHub：采集参数和 TikHub API Key。
- 模型供应商：用户选择的记录字段和模型供应商 API Key。
- 自定义模型 endpoint：用户选择的记录字段、Base URL 请求和对应密钥。
- Webhook 目标：用户启用的任务摘要。

### 12.4 本地文件安全

- 工作区外路径必须经过用户选择或后端白名单。
- 导出路径必须可写且用户可访问。
- 备份默认不包含密钥和大体积原始响应。
- 临时文件清理前要避免误删用户保存文件。
- WebView 只加载 asset protocol scope 内的文件。

### 12.5 合规声明

产品必须在报告和帮助文档中说明：

- 只采集公开数据。
- 不提供绕过登录、验证码或权限限制的能力。
- 报告实际包含模型生成内容时，必须说明其可能存在遗漏、误判或表达偏差。
- 用户应自行确认后续存储、传播和使用行为符合适用法律法规及平台规则。

## 15. 未覆盖风险

### 15.1 外部 API 细节风险

本文档定义了 TikHub 适配层和白名单机制，但具体 endpoint、字段名、分页规则、费用口径、限流响应仍需以 TikHub 当前官方接口为准。实现时必须逐个 endpoint 校准，不得用猜测字段直接落库。

### 15.2 模型供应商兼容风险

OpenAI-compatible endpoint 的实际兼容程度差异较大。即使协议名相同，不同供应商在结构化输出、错误格式、Token 统计、超时和流式响应上仍可能不同，需要通过适配层和回归样例逐步收敛。

### 15.3 PDF 渲染风险

中文字体、分页、长表格、链接和图表在 PDF 中可能反复出现布局问题。MVP 应限制 PDF 版式复杂度，优先保证中文、证据引用和基础分页稳定。

### 15.4 本地大数据量风险

大规模评论采集、长文本 AI 整理和多工作表导出会带来本地磁盘、内存和 SQLite 查询压力。实现时需要分页查询、分批写入、WAL、索引、导出流式写入和临时文件清理。

### 15.5 安全配置漂移风险

Tauri capabilities、`tauri-plugin-fs` 权限和 asset protocol scope 容易在开发中被临时放宽。实现时应把权限检查纳入本地测试和发布前检查，避免把 `$HOME/**/*` 一类宽权限带入发布版本。

### 15.6 口径漂移风险

国家/地区、标准化国家代码、来源、置信度和校验状态必须作为同一业务概念的不同属性处理。后续新增字段时要避免把同一概念拆成并列筛选字段，造成报告、导出和任务筛选口径冲突。
