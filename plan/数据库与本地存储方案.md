# 数据库与本地存储方案

版本：v0.2
日期：2026-07-17
关联 PRD：`智能数据整理平台_PRD.md` v0.7
文档定位：定义 SQLite、本地文件系统、工作区私有 API 配置 JSON、备份、迁移、索引、数据保留和反数据 slop 所需的数据模型

> 决策覆盖：从 v0.2 起，用户确认的统一 JSON API 配置方案覆盖本文及旧文档中“API Key 必须进入 macOS Keychain／系统安全存储”的既有口径。旧 Keychain 条目不由新版本读取或删除。

## 1. 存储目标

MVP 是本地优先产品，数据库和文件存储必须支撑以下目标：

- 单用户本地工作区。
- TikHub 与 AI API 配置以工作区私有 JSON 为唯一主数据源。
- API Key 以明文进入私有 JSON，但不进入 SQLite、日志、导出、Webhook 或应用备份。
- AI 配置本次只支持保存、完整性校验、查看和切换，不发起真实模型请求；当前自然语言计划仍使用 `local-rule-engine/rule-parser-v1`。
- 原始响应、标准化数据、AI 结果、证据、报告和导出文件可追溯。
- 支持任务失败、部分成功、待人工确认等真实状态。
- 支持 Excel 和 PDF 导出。
- 支持提示词版本、运行快照和基础回归样例。
- 支持手动备份。
- 断网时可以打开已有工作区和历史报告。

## 2. 存储分层

```text
工作区私有 API 配置注册表
  - secrets/api-config.json
  - 多条 TikHub 命名配置与当前配置 ID
  - 多条 AI 命名配置与当前配置 ID
  - 与配置对象分离的明文 credentials
  - 目录权限 0700，文件权限 0600

SQLite
  - 工作区
  - 任务
  - 结构化索引
  - 标准化数据
  - AI 运行记录
  - 提示词版本
  - 字段级证据
  - 报告和导出索引
  - 审计日志
  - API 配置的可重建镜像、密钥引用元数据和无明文运行快照

本地文件系统
  - 大体积原始响应
  - Excel 文件
  - PDF 文件
  - 备份包
  - 临时文件
  - 脱敏日志
```

## 3. SQLite 配置

建议初始化时执行：

```sql
PRAGMA foreign_keys = ON;
PRAGMA journal_mode = WAL;
PRAGMA wal_autocheckpoint = 1000;
PRAGMA synchronous = NORMAL;
PRAGMA temp_store = MEMORY;
```

说明：

- `foreign_keys = ON`：保证任务、记录、AI 运行和报告关系完整。
- `journal_mode = WAL`：提升读写并发，任务运行时前端仍能读取数据。
- `wal_autocheckpoint`：控制 WAL 文件增长。
- `synchronous = NORMAL`：在本地应用中平衡性能和可靠性。
- `temp_store = MEMORY`：降低临时查询磁盘压力。

维护命令：

```sql
PRAGMA optimize;
PRAGMA quick_check;
PRAGMA wal_checkpoint(TRUNCATE);
```

触发场景：

- 应用空闲时执行 `PRAGMA optimize`。
- 用户执行本地健康检查时执行 `quick_check`。
- 备份前可执行 WAL checkpoint。

## 4. 工作区目录结构

建议每个工作区一个目录：

```text
workspace/
  app.sqlite
  app.sqlite-wal
  app.sqlite-shm
  secrets/
    api-config.json
  raw/
    tikhub/
      {task_id}/
  exports/
    excel/
    pdf/
  reports/
    {report_id}/
  prompts/
    snapshots/
  logs/
  temp/
  backups/
```

目录规则：

- `secrets/` 固定为 `0700`，`api-config.json` 固定为 `0600`；拒绝符号链接、非普通文件和无法落实私有权限的文件系统。
- `api-config.json` 通过同目录临时文件、文件同步和原子重命名写入，进程内读写由同一互斥锁串行化。
- `raw/` 保存大体积原始响应文件，数据库只保存路径、哈希和摘要。
- `exports/` 保存用户可访问的 Excel 和 PDF 文件。
- `reports/` 保存报告数据模型快照。
- `prompts/snapshots/` 保存提示词版本快照，便于历史复现。
- `temp/` 可清理。
- `backups/` 保存用户手动备份。

### 5.2 secret_ref

用途：作为从 `api-config.json` 确定性生成的 SQLite 密钥引用镜像，不保存明文，也不是密钥主数据源。

字段建议：

- `id`
- `provider_type`：tikhub、model_provider、webhook。
- `provider_id`
- `secret_store_key`
- `masked_hint`
- `created_at`
- `updated_at`
- `last_tested_at`
- `last_test_status`
- `credential_revision`

规则：

- `secret_store_key` 只指向 `api-config.json#credentials/{credential_id}`，不得包含明文密钥。
- JSON 中的 `credential` 保存 `provider_type`、所属配置 ID、修订号和明文密钥；配置对象本身不重复保存密钥。
- 不保存完整 API Key。
- SQLite 镜像可以从 JSON 完整重建，不得反向覆盖 JSON。
- 删除当前配置后当前指针置空，不自动选择其他配置；后续排队任务等待用户显式选择新的当前配置。

## 8. 文件存储策略

### 8.1 原始响应

原始响应可能体积较大，不直接存入 SQLite。

保存策略：

- 文件路径：`raw/tikhub/{task_id}/{step_id}_{batch_id}.json`
- 数据库保存路径、哈希、大小、摘要。
- 可选压缩。
- 原始响应可由用户选择是否进入备份。

### 8.2 导出文件

Excel：

- 路径：`exports/excel/{task_id}_{report_id}.xlsx`
- 保存哈希、大小、生成时间。

PDF：

- 路径：`exports/pdf/{task_id}_{report_id}.pdf`
- 保存哈希、大小、页数、生成时间。

### 8.3 临时文件

临时文件：

- PDF 渲染中间文件。
- 大任务分批文件。
- 下载或预览缓存。

清理规则：

- 应用启动时清理过期临时文件。
- 导出成功后清理中间文件。
- 任务失败保留必要诊断摘要，不保留敏感原始中间文件。

## 9. 备份策略

MVP 支持手动备份。

默认包含：

- SQLite 数据库。
- 报告数据模型。
- Excel 和 PDF 导出索引。
- 提示词模板和版本。
- 脱敏日志。

默认不包含：

- `secrets/api-config.json` 和任何 API Key。
- 明文 Header。
- 临时缓存。
- 大体积原始响应。

用户可选：

- 是否包含原始响应文件。
- 是否包含导出文件。

备份前：

- 执行 WAL checkpoint。
- 检查数据库完整性。
- 写入备份 manifest。

备份 manifest：

```text
backup_version
created_at
app_version
schema_version
included_sections
file_hashes
```

## 10. 数据迁移

### 10.1 迁移原则

- 每次 Schema 变更必须有迁移版本。
- 迁移必须可重复检测。
- 迁移前自动备份数据库。
- 迁移失败必须回滚或提示用户恢复备份。
- 数据库迁移不得读取或删除 Keychain 条目，避免触发系统权限窗。

### 10.2 迁移表

```text
schema_migrations
  version
  name
  applied_at
  checksum
```

### 10.3 兼容策略

- 新版本可读取旧版本工作区并执行迁移。
- 首次没有 JSON 时，只从旧 SQLite 导入 TikHub、模型供应商、默认模型、名称、Base URL、测试状态和密钥引用等非敏感元数据。
- 旧凭据统一标记为 `needs_rebind`，TikHub 与 AI 当前指针均置空；用户必须在对应管理弹窗重新输入一次 Token 或 API Key。
- 旧 Keychain 条目保留，确认新版本正常后由用户在“钥匙串访问”中手动删除；旧版本回滚时仍可继续使用。
- 不支持降级写入旧版本数据库。

### 10.4 工作区 Schema v8

- v8 迁移前使用 SQLite `VACUUM INTO` 在 `backups/` 创建一致性备份。
- `backups/` 目录权限为 `0700`，备份文件权限为 `0600`。
- 迁移只删除从未领取、没有检查点且不是恢复状态的新鲜排队任务旧连接快照，使其在首次领取时绑定当前 TikHub 配置。
- 运行中、恢复中和历史任务的不可变快照必须保留。
- 回滚旧应用前必须停止 Sortlytic，再恢复该 v8 前备份。

## 11. 数据保留与清理

可清理：

- 临时文件。
- 过期任务日志。
- 已导出的中间文件。
- 失败任务的无敏感中间缓存。

不可自动清理：

- 用户保存的任务。
- 报告。
- 导出文件索引。
- 提示词版本。
- AI 运行记录。
- 字段级证据。

用户可手动清理：

- 原始响应文件。
- 历史运行日志。
- 导出文件。
- 备份包。

清理前必须二次确认。

## 12. 隐私与安全

### 12.1 密钥

- API Key 以明文保存在当前工作区 `secrets/api-config.json`，目录权限为 `0700`，文件权限为 `0600`。
- 私有权限只能限制其他本机用户，不能防止同一 macOS 账户下的恶意进程读取 JSON。
- SQLite、日志、Excel、PDF、Webhook 和应用备份不保存完整 API Key。
- JSON 无法解析、Schema 版本未知、路径是符号链接或权限不安全时失败关闭，不覆盖原文件；API 配置管理和新任务领取不可继续，但已有工作区和历史数据仍可浏览。
- 旧 Keychain 条目不自动读取或删除；旧配置必须重新输入一次凭据后才能激活。
- 被运行中或恢复中不可变快照引用的 TikHub 配置不得编辑或删除，但允许切换到其他配置。

### 12.2 日志

- 日志保存脱敏请求信息。
- 不保存完整 Header。
- 不保存 Authorization。
- 错误堆栈写入前必须过滤敏感字符串。

### 12.3 导出

- Excel 不包含 API Key。
- PDF 不包含 API Key。
- Webhook 不包含 API Key。
- 应用备份不包含 `secrets/api-config.json`。
- 报告中包含 AI 免责声明和数据来源声明。

## 13. 数据库验收标准

- 新建工作区后自动创建 SQLite 数据库和目录结构。
- 外键约束启用。
- WAL 模式启用。
- 一个 JSON 可同时保存多条 TikHub 与多条 AI 配置，且每类最多一条当前配置。
- API Key 只出现在权限正确的 `api-config.json`，不出现在 SQLite、审计、日志、导出、Webhook 或备份。
- 创建任务、采集数据、AI 整理、生成报告、导出文件后，关键关系可追溯。
- 删除当前配置后当前指针为空；新鲜排队任务等待新的当前 TikHub 配置。
- 运行中与恢复任务切换当前配置后仍使用原运行快照。
- 提示词版本变更后，旧任务仍能看到原运行版本。
- 核心洞察缺少 `field_provenance` 时无法进入核心导出。
- 备份文件不包含明文密钥。

## 14. 测试计划

### 14.1 单元测试

- Schema 创建。
- 外键约束。
- 索引存在。
- JSON 重新加载多条 TikHub、AI 配置和凭据无损。
- 私有目录与文件权限分别为 `0700`、`0600`。
- 符号链接、宽松权限、非法版本和损坏 JSON 均失败关闭。
- 原子写入中断不破坏旧文件，并发保存不丢配置。
- 密钥引用、审计和 SQLite 镜像不含明文。
- 提示词版本唯一约束。
- 字段级证据校验。

### 14.2 集成测试

- 创建工作区。
- 旧 SQLite 元数据导入后保留名称、端点、模型和引用，统一为 `needs_rebind`，且不调用 Keychain。
- 第一条有效配置自动激活，后续配置可显式切换；同一 AI 供应商可保存多个账号或端点。
- 创建任务。
- 保存采集计划。
- 保存原始记录和标准化记录。
- 保存 AI 运行和运行快照。
- 保存字段级证据。
- 生成报告和导出记录。
- 备份和恢复。
- 新鲜任务领取时绑定当前 TikHub 配置，运行中与恢复任务继续使用旧快照。

### 14.3 异常测试

- 数据库文件被占用。
- 磁盘空间不足。
- WAL 文件异常。
- 迁移失败。
- 原始响应文件缺失。
- 导出文件路径不可写。
- 密钥引用失效。
- JSON 缺失、损坏、版本未知或无当前 TikHub 配置。
- 当前配置被运行中或恢复中快照引用时尝试编辑、删除。
