# 数据库与本地存储方案

版本：v0.1  
日期：2026-07-04  
关联 PRD：`智能数据整理平台_PRD.md` v0.5  
文档定位：定义 SQLite、本地文件系统、安全存储、备份、迁移、索引、数据保留和反数据 slop 所需的数据模型

## 1. 存储目标

MVP 是本地优先产品，数据库和文件存储必须支撑以下目标：

- 单用户本地工作区。
- API Key 不进入数据库。
- 原始响应、标准化数据、AI 结果、证据、报告和导出文件可追溯。
- 支持任务失败、部分成功、待人工确认等真实状态。
- 支持 Excel 和 PDF 导出。
- 支持提示词版本、运行快照和基础回归样例。
- 支持手动备份。
- 断网时可以打开已有工作区和历史报告。

## 2. 存储分层

```text
系统安全存储
  - TikHub API Key
  - 模型供应商 API Key
  - Webhook 密钥或 Header 中的敏感字段

SQLite
  - 工作区
  - 任务
  - 结构化索引
  - 标准化数据
  - AI 运行记录
  - 提示词版本
  - 字段级证据
  - 报告和导出索引
  - 审计日志

本地文件系统
  - 大体积原始响应
  - Excel 文件
  - PDF 文件
  - 备份包
  - 临时文件
  - 脱敏日志
```

## 3. SQLite 配置

建议初始化时执行：

```sql
PRAGMA foreign_keys = ON;
PRAGMA journal_mode = WAL;
PRAGMA wal_autocheckpoint = 1000;
PRAGMA synchronous = NORMAL;
PRAGMA temp_store = MEMORY;
```

说明：

- `foreign_keys = ON`：保证任务、记录、AI 运行和报告关系完整。
- `journal_mode = WAL`：提升读写并发，任务运行时前端仍能读取数据。
- `wal_autocheckpoint`：控制 WAL 文件增长。
- `synchronous = NORMAL`：在本地应用中平衡性能和可靠性。
- `temp_store = MEMORY`：降低临时查询磁盘压力。

维护命令：

```sql
PRAGMA optimize;
PRAGMA quick_check;
PRAGMA wal_checkpoint(TRUNCATE);
```

触发场景：

- 应用空闲时执行 `PRAGMA optimize`。
- 用户执行本地健康检查时执行 `quick_check`。
- 备份前可执行 WAL checkpoint。

## 4. 工作区目录结构

建议每个工作区一个目录：

```text
workspace/
  app.sqlite
  app.sqlite-wal
  app.sqlite-shm
  raw/
    tikhub/
      {task_id}/
  exports/
    excel/
    pdf/
  reports/
    {report_id}/
  prompts/
    snapshots/
  logs/
  temp/
  backups/
```

目录规则：

- `raw/` 保存大体积原始响应文件，数据库只保存路径、哈希和摘要。
- `exports/` 保存用户可访问的 Excel 和 PDF 文件。
- `reports/` 保存报告数据模型快照。
- `prompts/snapshots/` 保存提示词版本快照，便于历史复现。
- `temp/` 可清理。
- `backups/` 保存用户手动备份。

## 8. 文件存储策略

### 8.1 原始响应

原始响应可能体积较大，不直接存入 SQLite。

保存策略：

- 文件路径：`raw/tikhub/{task_id}/{step_id}_{batch_id}.json`
- 数据库保存路径、哈希、大小、摘要。
- 可选压缩。
- 原始响应可由用户选择是否进入备份。

### 8.2 导出文件

Excel：

- 路径：`exports/excel/{task_id}_{report_id}.xlsx`
- 保存哈希、大小、生成时间。

PDF：

- 路径：`exports/pdf/{task_id}_{report_id}.pdf`
- 保存哈希、大小、页数、生成时间。

### 8.3 临时文件

临时文件：

- PDF 渲染中间文件。
- 大任务分批文件。
- 下载或预览缓存。

清理规则：

- 应用启动时清理过期临时文件。
- 导出成功后清理中间文件。
- 任务失败保留必要诊断摘要，不保留敏感原始中间文件。

## 9. 备份策略

MVP 支持手动备份。

默认包含：

- SQLite 数据库。
- 报告数据模型。
- Excel 和 PDF 导出索引。
- 提示词模板和版本。
- 脱敏日志。

默认不包含：

- API Key。
- 明文 Header。
- 临时缓存。
- 大体积原始响应。

用户可选：

- 是否包含原始响应文件。
- 是否包含导出文件。

备份前：

- 执行 WAL checkpoint。
- 检查数据库完整性。
- 写入备份 manifest。

备份 manifest：

```text
backup_version
created_at
app_version
schema_version
included_sections
file_hashes
```

## 10. 数据迁移

### 10.1 迁移原则

- 每次 Schema 变更必须有迁移版本。
- 迁移必须可重复检测。
- 迁移前自动备份数据库。
- 迁移失败必须回滚或提示用户恢复备份。

### 10.2 迁移表

```text
schema_migrations
  version
  name
  applied_at
  checksum
```

### 10.3 兼容策略

- 新版本可读取旧版本工作区并执行迁移。
- 迁移前显示风险提示。
- 不支持降级写入旧版本数据库。

## 11. 数据保留与清理

可清理：

- 临时文件。
- 过期任务日志。
- 已导出的中间文件。
- 失败任务的无敏感中间缓存。

不可自动清理：

- 用户保存的任务。
- 报告。
- 导出文件索引。
- 提示词版本。
- AI 运行记录。
- 字段级证据。

用户可手动清理：

- 原始响应文件。
- 历史运行日志。
- 导出文件。
- 备份包。

清理前必须二次确认。

## 12. 隐私与安全

### 12.1 密钥

- API Key 只存在系统安全存储。
- SQLite 不保存明文密钥。
- 备份不包含密钥。
- 删除密钥后相关连接测试失败，相关任务不可继续。

### 12.2 日志

- 日志保存脱敏请求信息。
- 不保存完整 Header。
- 不保存 Authorization。
- 错误堆栈写入前必须过滤敏感字符串。

### 12.3 导出

- Excel 不包含 API Key。
- PDF 不包含 API Key。
- Webhook 不包含 API Key。
- 报告中包含 AI 免责声明和数据来源声明。

## 13. 数据库验收标准

- 新建工作区后自动创建 SQLite 数据库和目录结构。
- 外键约束启用。
- WAL 模式启用。
- API Key 不出现在 SQLite。
- 创建任务、采集数据、AI 整理、生成报告、导出文件后，关键关系可追溯。
- 删除密钥后依赖任务不可继续运行。
- 提示词版本变更后，旧任务仍能看到原运行版本。
- 核心洞察缺少 `field_provenance` 时无法进入核心导出。
- 备份文件不包含明文密钥。

## 14. 测试计划

### 14.1 单元测试

- Schema 创建。
- 外键约束。
- 索引存在。
- 密钥引用不含明文。
- 提示词版本唯一约束。
- 字段级证据校验。

### 14.2 集成测试

- 创建工作区。
- 创建任务。
- 保存采集计划。
- 保存原始记录和标准化记录。
- 保存 AI 运行和运行快照。
- 保存字段级证据。
- 生成报告和导出记录。
- 备份和恢复。

### 14.3 异常测试

- 数据库文件被占用。
- 磁盘空间不足。
- WAL 文件异常。
- 迁移失败。
- 原始响应文件缺失。
- 导出文件路径不可写。
- 密钥引用失效。
