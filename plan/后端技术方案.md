# 后端技术方案

版本：v0.2
日期：2026-07-17
关联 PRD：`智能数据整理平台_PRD.md` v0.7
文档定位：定义 Tauri 本地后端、任务队列、外部 API 调用、模型适配、提示词运行、导出、安全和错误处理

## 1. 后端定位

本项目 MVP 不做云端服务器。这里的“后端”指运行在用户本机的 Tauri Rust 后端和本地服务层。

后端职责：

- 管理工作区私有 JSON API 配置注册表、SQLite 派生镜像和本地文件。
- 统一执行 TikHub REST API 请求。
- 保存、校验和切换 AI API 配置；当前版本不执行模型供应商 API 请求。
- 使用本地规则引擎生成自然语言采集计划。
- 管理任务队列和运行日志。
- 管理提示词模板、Schema 校验和基础回归样例。
- 生成 Excel、PDF 和 Webhook 摘要。
- 控制权限、安全、脱敏和错误边界。

非目标：

- 不提供云端 API 代理。
- 不做自研爬虫。
- 不做云端长任务。
- 不做团队协作后端。
- 不把 TikHub API Key 或模型 API Key 暴露给前端和 AI prompt。

## 2. 技术选型

推荐：

- Tauri v2 Rust 后端。
- Rust async runtime。
- Rust HTTP client。
- SQLite 访问层。
- 私有 JSON 注册表与原子文件写入。
- 文件系统 API。
- 后端 Schema 校验。

补充：

- Excel 和 PDF 均由 Rust 后端导出器生成，前端只触发导出任务并展示进度、错误和结果路径。

## 3. 后端分层

```text
Tauri Commands
  |
  v
应用服务层
  - 工作区、API 配置注册表、任务、AI、导出、Webhook
  |
  v
领域服务层
  - TikHub、本地规则、提示词、Schema、报告、成本、证据；模型适配为后续能力
  |
  v
基础设施层
  - 私有 JSON、SQLite 派生镜像、文件系统、HTTP、日志、时间、队列
```

## 4. 模块划分

### 4.1 Commands

API 配置使用独立命令边界：

```text
get_api_profile_registry
save_api_profile
test_api_profile
activate_api_profile
delete_api_profile
```

命令由 `api_profile_commands.rs` 与服务层实现，注册表类型、存储和 SQLite 镜像分别位于 `api_profiles.rs`、`api_profiles/storage.rs` 和 `api_profiles/mirror.rs`。前端完成迁移后，旧密钥、TikHub 单例和模型供应商外部变更命令应从 invoke handler 移除，防止绕过 JSON 主数据源。

命令设计原则：

- 命令入参和返回值必须可序列化。
- 命令必须返回统一错误结构。
- 敏感命令必须做权限和状态检查。
- 前端不能绕过命令直接访问密钥、数据库和外部 API。

统一错误结构：

```text
AppError
  code
  message
  stage
  retryable
  safe_details
```

`safe_details` 必须脱敏。

### 4.2 Workspace Service

职责：

- 初始化工作区。
- 解析工作区路径。
- 创建数据库和目录结构。
- 计算本地存储占用。
- 创建备份和恢复备份。

目录建议：

```text
workspace/
  app.sqlite
  secrets/
    api-config.json
  raw/
  exports/
    excel/
    pdf/
  reports/
  temp/
  backups/
  logs/
```

默认工作区的配置文件位于 `~/Library/Application Support/com.steven.sortlytic/default-workspace/secrets/api-config.json`。`secrets` 目录固定为 `0700`，文件固定为 `0600`；配置文件不进入应用备份、导出或前端文件访问范围。

工作区 Schema v8 迁移前在 `backups` 目录创建权限为 `0600` 的 SQLite 一致性备份；迁移只清除从未领取的排队任务旧连接快照，运行中、恢复中和历史快照均保留。

### 4.3 API Profile Registry Service

职责：

- 在同一个 `api-config.json` 中保存多条命名 TikHub 与 AI 配置、每类当前配置 ID 以及凭据对象。
- 注册表固定使用 `schema_version: 1`、`active_profile_ids`、`tikhub_profiles`、`ai_profiles` 和 `credentials` 五个顶层数据域。
- 以稳定配置 UUID 和密钥引用 ID 关联配置与凭据，配置对象本身不重复保存密钥。
- 新建、编辑、测试、切换和删除配置，并返回仅含首尾脱敏密钥的安全视图。
- 将 JSON 确定性投影为 SQLite 运行镜像；SQLite 只保存可重建的非敏感元数据。
- 使用进程内互斥锁串行化读写，并通过同目录临时文件、文件同步和原子重命名更新注册表。

要求：

- JSON 是 TikHub 与 AI API 配置的唯一主数据源，`credentials` 中的密钥以明文保存。
- 拒绝符号链接、非普通文件、宽松权限、未知 Schema 版本和损坏 JSON；读取失败时不得覆盖原文件。
- 无法落实 `0700`/`0600` 私有权限的文件系统拒绝保存；注册表异常只阻断 API 管理和新任务领取，本地工作区浏览与历史数据仍可访问。
- SQLite 不保存明文密钥。
- 前端不接收明文密钥。
- 日志不输出完整密钥。
- 备份默认不包含密钥。
- 旧 Keychain 只作为迁移历史保留：迁移过程不读取也不删除旧条目，因此不会触发系统权限窗。
- 从旧 SQLite 导入的配置保留名称、端点、模型和引用，但统一标记为 `needs_rebind`，当前指针置空，用户需重新输入一次密钥。

### 4.4 Provider Service

职责：

- 管理 AI 配置注册表和当前配置指针。
- 管理自定义 OpenAI-compatible endpoint。
- 校验供应商类型、协议格式、Base URL、默认模型和密钥完整性。
- 支持 OpenAI、Anthropic、Gemini、自定义 OpenAI-compatible 与 Ollama，同一供应商可保存多条账号或端点配置。
- API 格式固定为 `openai_compatible`、`anthropic_messages`、`gemini` 或 `ollama`。

关键规则：

- 不把供应商逻辑写死为 OpenAI、Gemini、Claude。
- 供应商和协议格式分离。
- 模型唯一标识使用 `provider_id + model_id`。
- Ollama 可不提供密钥，其余类型必须满足各自完整性约束。
- 当前 `test_api_profile` 只执行本地配置校验，不发真实模型请求；界面和日志不得将其表述为模型连通测试。
- 第一条校验成功的 AI 配置可自动成为当前配置，后续配置由用户显式切换；删除当前配置后指针置空，不自动选择替代项。

### 4.5 TikHub Service

职责：

- 多条 TikHub 命名配置管理与真实账户连通测试。
- 端点只接受 `https://api.tikhub.io` 或 `https://api.tikhub.dev`。
- 平台能力注册。
- 数据类型能力注册。
- 参数白名单校验。
- 成本估算。
- REST API 请求。
- 限流识别。
- 失败重试。
- 响应摘要。

MVP 平台：

- TikTok
- 抖音
- 小红书

请求要求：

- 所有 TikHub 请求必须经过统一请求层。
- 只有测试状态为 `success` 的配置才能激活；测试失败的配置保留为 `failed` 以便编辑重试。
- 每类最多一条当前配置，切换当前配置不修改旧配置内容。
- 用户确认前不得发起会产生正式采集费用的请求。
- 本地规则引擎或未来 AI 生成的参数都必须经过白名单和 Schema 校验。
- 请求日志必须脱敏。

### 4.6 AI Service（当前边界）

职责：

- 使用本地规则引擎生成自然语言采集计划。
- 将解析结果交给现有 Schema 与业务白名单校验。
- 保存、查看、校验和切换 AI API 配置，作为未来模型执行的准备能力。

当前运行步骤：

```text
读取任务输入
  -> 本地规则引擎解析
  -> Schema 校验
  -> 用户确认结构化计划
  -> 进入 TikHub 任务队列
```

当前 AI 配置不会被自然语言计划或任务执行器调用。真实模型调用、`ai_run`、模型 Runtime Snapshot、字段级 AI 证据和模型错误处理属于后续能力，接入时必须同时落实提示词版本、输出 Schema、证据引用和基础回归样例，不能只增加 HTTP 请求。

### 4.7 Prompt Service

MVP 必做。

职责：

- 管理提示词模板。
- 管理提示词版本。
- 管理提示词变更说明。
- 绑定输出 Schema。
- 运行基础回归样例。
- 支持回滚。

提示词类型：

- 采集计划生成。
- 缺失条件追问。
- 通用摘要。
- 评论情绪分析。
- 主题分类。
- 实体抽取。
- 代表性观点提取。
- 内容爆点拆解。
- 风险与异常提示。
- 平台差异对比。

基础回归样例至少覆盖：

- 正常自然语言需求。
- 缺少平台。
- 缺少国家 / 地区。
- 缺少时间范围。
- 多平台高成本任务。
- 证据不足。
- 字段冲突。
- 模型输出非 JSON。

### 4.8 Job Service

职责：

- 创建任务运行。
- 队列调度。
- 取消、失败重试、复制。
- 分阶段保存。
- 写运行日志。
- 计算实际成本。

TikHub 配置绑定规则：

- 新任务入队时不冻结 TikHub 配置；执行器首次领取任务时，在同一事务中读取当时的当前 TikHub 配置并创建不可变运行快照。
- 已经运行过并进入恢复队列的任务复用原快照，不跟随当前配置切换。
- 快照只保存配置 ID、修订号、Base URL 和密钥引用等运行元数据，不保存明文密钥；运行时按引用从 JSON 读取凭据。
- JSON 缺失、损坏、无当前 TikHub 配置、当前配置未验证或镜像不一致时，领取事务失败关闭，任务保持排队。
- 被运行中或恢复中快照引用的 TikHub 配置禁止编辑和删除，但允许切换到另一配置。

任务阶段：

- 参数校验。
- 成本估算。
- 用户确认。
- TikHub 采集。
- 标准化。
- AI 整理（真实模型执行接入后启用）。
- 证据生成。
- 报告生成。
- 导出。
- Webhook。

任务取消规则：

- 取消后不得发起新的外部请求。
- 已发出的请求等待完成或超时。
- 保存取消时间和已完成阶段。

### 4.9 Export Service

职责：

- 导出前完整性检查。
- 生成 Excel 工作簿。
- 生成 PDF 报告。
- 写入导出文件。
- 保存导出任务状态。
- 返回本地路径。

完整性检查：

- 报告数据模型存在。
- 核心洞察有证据引用。
- 待人工确认字段已标记。
- PDF 字体可用。
- Excel 工作表完整。
- 不包含 API Key。

### 4.10 Webhook Service

职责：

- 保存 Webhook 配置。
- 测试 Webhook。
- 任务完成后发送摘要。
- 记录成功和失败。

Webhook 内容：

- 任务 ID。
- 任务名称。
- 平台。
- 状态。
- 数据量。
- 核心摘要。
- 导出文件路径或用户可访问的本地路径说明。
- 生成时间。

禁止：

- API Key。
- 完整 Header。
- 未脱敏错误日志。
- 超大原始 JSON。

## 5. 模型适配层（后续能力）

### 5.1 目标

统一不同模型供应商和协议格式，让上层 AI Service 不直接关心具体供应商细节。本节描述接入真实模型时的目标态，不代表当前版本已经发起模型请求。

### 5.2 支持协议

MVP 建议：

- OpenAI-compatible。
- Gemini 官方或兼容格式。
- Anthropic Messages。

### 5.3 适配层输入

```text
ModelRequest
  provider_id
  model_id
  api_format
  base_url
  secret_ref
  prompt_version
  input_messages

  retry_count

```

### 5.5 错误分类

- 网络错误。
- 认证错误。
- 限流错误。

- 协议不兼容。
- 非 JSON 输出。

- 超时。

## 6. 数据标准化与证据

### 6.1 标准化职责

不同平台字段不同，后端要统一生成标准化记录：

- 平台。
- 原始记录 ID。
- 原始链接。
- 作者或账号。
- 发布时间。
- 正文或标题。
- 标签。
- 互动指标。
- 评论数。
- 国家 / 地区信息。
- 采集时间。

### 6.2 字段级证据

真实模型执行接入后，AI 生成字段必须保存：

- 目标字段。
- 生成值。
- 来源记录 ID。
- 来源字段。
- 转换理由。
- 置信度。
- 校验状态。
- 模型运行 ID。
- 提示词版本。
- 人工审核状态。

## 7. 安全与脱敏

API 配置安全边界：

- 密钥以明文写入当前工作区私有 `secrets/api-config.json`，目录权限为 `0700`，文件权限为 `0600`。
- `0600` 只能限制其他本机用户，不能防止同一 macOS 账户下的恶意进程读取；这是停止使用 Keychain 后明确接受的安全降级。
- SQLite、审计、错误、日志、导出、Webhook 和运行快照均不得包含明文密钥。
- 旧 Keychain 条目不读取、不自动删除；确认新版本正常后，用户可自行在“钥匙串访问”中清理。

### 7.1 日志脱敏

必须脱敏：

- API Key。
- Authorization Header。
- Cookie。
- 自定义 Header 中疑似密钥字段。
- 完整错误堆栈中的敏感信息。

### 7.2 AI 输入控制

- TikHub API Key 不进入 prompt。
- 模型供应商 API Key 不进入 prompt。
- 只发送用户选择的记录字段给模型。
- 用户可选择是否发送正文、评论、作者、链接等字段。

### 7.3 导出控制

- Excel 不包含 API Key。
- PDF 不包含 API Key。
- Webhook 不包含 API Key。
- 备份默认不包含 API Key。

## 8. 性能策略

### 8.1 大任务

- 分页采集。
- 分批写入数据库。
- 大体积原始响应写文件。
- 任务阶段性提交。
- UI 只读取摘要和分页数据。

### 8.2 数据库

- 使用 WAL。
- 关键查询建索引。
- 大表分页。
- 定期执行优化。

### 8.3 外部请求

- 超时。
- 重试。
- 限流退避。
- 并发上限。
- 成本上限。

## 9. Tauri 权限边界

Tauri 后端必须最小化暴露命令：

- 前端只能调用声明过的 commands。
- 文件访问限制在工作区目录和用户选择的导出路径。
- 前端不能直接读取 `secrets/api-config.json`，API 配置命令只返回脱敏安全视图。
- 网络请求集中在后端请求层。
- 移动端平台能力通过插件封装，不在前端直接访问。

- 密钥哨兵只出现在 JSON，不进入 SQLite、审计、错误、日志、导出或安全视图。
- 参数白名单。

## 11. 后端验收标准

- TikHub 与 AI 多条命名配置可保存、查看和切换，完整密钥只存在于私有 JSON。
- TikHub 真实账户连接测试可用；AI 与自定义 OpenAI-compatible 当前只做本地完整性校验。
- 旧 Keychain 不再被运行时或迁移读取，不再触发钥匙串权限申请；旧配置以 `needs_rebind` 等待重输。
- 第一条有效配置可自动激活，未验证配置不能激活，删除当前配置后指针为空。
- 新鲜任务首次领取时绑定当前 TikHub，恢复任务复用不可变快照。
- 自然语言采集计划由本地规则引擎生成，不调用已保存的 AI 当前配置。
- 自然语言采集计划必须经过 Schema 和白名单校验。
- 用户确认前不发起正式 TikHub 采集。
- 后续真实 AI 运行接入后，运行记录必须包含 Runtime Snapshot，核心 AI 字段必须包含来源证据。
- 导出前完整性检查能阻止无证据核心洞察。
- 日志、导出、Webhook、备份不泄露 API Key。
- 任务失败时记录失败阶段、原因、可重试性和下一步。
