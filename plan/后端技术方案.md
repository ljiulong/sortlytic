# 后端技术方案

版本：v0.1  
日期：2026-07-04  
关联 PRD：`智能数据整理平台_PRD.md` v0.7
文档定位：定义 Tauri 本地后端、任务队列、外部 API 调用、模型适配、提示词运行、导出、安全和错误处理

## 1. 后端定位

本项目 MVP 不做云端服务器。这里的“后端”指运行在用户本机的 Tauri Rust 后端和本地服务层。

后端职责：

- 管理系统安全存储。
- 管理 SQLite 和本地文件。
- 统一执行 TikHub REST API 请求。
- 统一执行模型供应商 API 请求。
- 管理任务队列和运行日志。
- 执行提示词模板、Schema 校验和基础回归样例。
- 生成 Excel、PDF 和 Webhook 摘要。
- 控制权限、安全、脱敏和错误边界。

非目标：

- 不提供云端 API 代理。
- 不做自研爬虫。
- 不做云端长任务。
- 不做团队协作后端。
- 不把 TikHub API Key 或模型 API Key 暴露给前端和 AI prompt。

## 2. 技术选型

推荐：

- Tauri v2 Rust 后端。
- Rust async runtime。
- Rust HTTP client。
- SQLite 访问层。
- 系统安全存储插件或平台原生安全存储封装。
- 文件系统 API。
- 后端 Schema 校验。

补充：

- Excel 和 PDF 均由 Rust 后端导出器生成，前端只触发导出任务并展示进度、错误和结果路径。

## 3. 后端分层

```text
Tauri Commands
  |
  v
应用服务层
  - 工作区、密钥、供应商、任务、AI、导出、Webhook
  |
  v
领域服务层
  - TikHub、模型适配、提示词、Schema、报告、成本、证据
  |
  v
基础设施层
  - SQLite、文件系统、安全存储、HTTP、日志、时间、队列
```

## 4. 模块划分

### 4.1 Commands

建议命令分组：

```text
commands/
  workspace.rs
  secrets.rs
  providers.rs
  tikhub.rs
  tasks.rs
  ai.rs
  prompts.rs
  records.rs
  reports.rs
  exports.rs
  webhooks.rs
  diagnostics.rs
```

命令设计原则：

- 命令入参和返回值必须可序列化。
- 命令必须返回统一错误结构。
- 敏感命令必须做权限和状态检查。
- 前端不能绕过命令直接访问密钥、数据库和外部 API。

统一错误结构：

```text
AppError
  code
  message
  stage
  retryable
  safe_details
```

`safe_details` 必须脱敏。

### 4.2 Workspace Service

职责：

- 初始化工作区。
- 解析工作区路径。
- 创建数据库和目录结构。
- 计算本地存储占用。
- 创建备份和恢复备份。

目录建议：

```text
workspace/
  app.sqlite
  raw/
  exports/
    excel/
    pdf/
  reports/
  temp/
  backups/
  logs/
```

### 4.3 Secret Service

职责：

- 保存密钥到系统安全存储。
- 返回 `secret_ref`。
- 读取密钥供后端请求层使用。
- 删除密钥。
- 掩码显示。

要求：

- SQLite 不保存明文密钥。
- 前端不接收明文密钥。
- 日志不输出完整密钥。
- 备份默认不包含密钥。

### 4.4 Provider Service

职责：

- 管理 Provider Registry。
- 管理模型供应商配置。
- 管理自定义 OpenAI-compatible endpoint。
- 执行连接测试。
- 维护模型列表、默认模型、能力标签和成本策略。

关键规则：

- 不把供应商逻辑写死为 OpenAI、Gemini、Claude。
- 供应商和协议格式分离。
- 模型唯一标识使用 `provider_id + model_id`。
- 连接测试要低成本。
- 失败时输出供应商、协议、状态码或网络错误摘要。

### 4.5 TikHub Service

职责：

- TikHub API Key 连接测试。
- 平台能力注册。
- 数据类型能力注册。
- 参数白名单校验。
- 成本估算。
- REST API 请求。
- 限流识别。
- 失败重试。
- 响应摘要。

MVP 平台：

- TikTok
- 抖音
- 小红书
- YouTube
- Instagram

请求要求：

- 所有 TikHub 请求必须经过统一请求层。
- 用户确认前不得发起会产生正式采集费用的请求。
- AI 生成参数必须经过白名单和 Schema 校验。
- 请求日志必须脱敏。

### 4.6 AI Service

职责：

- 执行自然语言采集计划生成。
- 执行 AI 整理模板。
- 调用模型适配层。
- 保存 `ai_run`。
- 保存 Runtime Snapshot。
- 保存字段级证据。
- 处理模型失败、非 JSON 输出、Schema 失败。

运行步骤：

```text
读取任务输入
  -> 选择提示词版本
  -> 选择供应商和模型
  -> 构造 Runtime Snapshot
  -> 调用模型适配层
  -> 解析结构化输出
  -> Schema 校验
  -> 保存 ai_run 和 field_provenance
  -> 返回可展示结果
```

### 4.7 Prompt Service

MVP 必做。

职责：

- 管理提示词模板。
- 管理提示词版本。
- 管理提示词变更说明。
- 绑定输出 Schema。
- 运行基础回归样例。
- 支持回滚。

提示词类型：

- 采集计划生成。
- 缺失条件追问。
- 通用摘要。
- 评论情绪分析。
- 主题分类。
- 实体抽取。
- 代表性观点提取。
- 内容爆点拆解。
- 风险与异常提示。
- 平台差异对比。

基础回归样例至少覆盖：

- 正常自然语言需求。
- 缺少平台。
- 缺少国家 / 地区。
- 缺少时间范围。
- 多平台高成本任务。
- 证据不足。
- 字段冲突。
- 模型输出非 JSON。

### 4.8 Job Service

职责：

- 创建任务运行。
- 队列调度。
- 取消、失败重试、复制。
- 分阶段保存。
- 写运行日志。
- 计算实际成本。

任务阶段：

- 参数校验。
- 成本估算。
- 用户确认。
- TikHub 采集。
- 标准化。
- AI 整理。
- 证据生成。
- 报告生成。
- 导出。
- Webhook。

任务取消规则：

- 取消后不得发起新的外部请求。
- 已发出的请求等待完成或超时。
- 保存取消时间和已完成阶段。

### 4.9 Export Service

职责：

- 导出前完整性检查。
- 生成 Excel 工作簿。
- 生成 PDF 报告。
- 写入导出文件。
- 保存导出任务状态。
- 返回本地路径。

完整性检查：

- 报告数据模型存在。
- 核心洞察有证据引用。
- 待人工确认字段已标记。
- PDF 字体可用。
- Excel 工作表完整。
- 不包含 API Key。

### 4.10 Webhook Service

职责：

- 保存 Webhook 配置。
- 测试 Webhook。
- 任务完成后发送摘要。
- 记录成功和失败。

Webhook 内容：

- 任务 ID。
- 任务名称。
- 平台。
- 状态。
- 数据量。
- 核心摘要。
- 导出文件路径或用户可访问的本地路径说明。
- 生成时间。

禁止：

- API Key。
- 完整 Header。
- 未脱敏错误日志。
- 超大原始 JSON。

## 5. 模型适配层

### 5.1 目标

统一不同模型供应商和协议格式，让上层 AI Service 不直接关心具体供应商细节。

### 5.2 支持协议

MVP 建议：

- OpenAI-compatible。
- Gemini 官方或兼容格式。
- Anthropic Messages。

### 5.3 适配层输入

```text
ModelRequest
  provider_id
  model_id
  api_format
  base_url
  secret_ref
  prompt_version
  input_messages
  output_schema
  temperature
  max_tokens
  timeout
```

### 5.4 适配层输出

```text
ModelResponse
  raw_output
  parsed_json
  input_tokens
  output_tokens
  latency_ms
  first_token_latency_ms
  retry_count
  provider_status
  error
```

### 5.5 错误分类

- 网络错误。
- 认证错误。
- 限流错误。
- 供应商不可用。
- 协议不兼容。
- 非 JSON 输出。
- Schema 校验失败。
- 超时。
- 成本上限触发。

## 6. 数据标准化与证据

### 6.1 标准化职责

不同平台字段不同，后端要统一生成标准化记录：

- 平台。
- 原始记录 ID。
- 原始链接。
- 作者或账号。
- 发布时间。
- 正文或标题。
- 标签。
- 互动指标。
- 评论数。
- 国家 / 地区信息。
- 采集时间。

### 6.2 字段级证据

AI 生成字段必须保存：

- 目标字段。
- 生成值。
- 来源记录 ID。
- 来源字段。
- 转换理由。
- 置信度。
- 校验状态。
- 模型运行 ID。
- 提示词版本。
- 人工审核状态。

## 7. 安全与脱敏

### 7.1 日志脱敏

必须脱敏：

- API Key。
- Authorization Header。
- Cookie。
- 自定义 Header 中疑似密钥字段。
- 完整错误堆栈中的敏感信息。

### 7.2 AI 输入控制

- TikHub API Key 不进入 prompt。
- 模型供应商 API Key 不进入 prompt。
- 只发送用户选择的记录字段给模型。
- 用户可选择是否发送正文、评论、作者、链接等字段。

### 7.3 导出控制

- Excel 不包含 API Key。
- PDF 不包含 API Key。
- Webhook 不包含 API Key。
- 备份默认不包含 API Key。

## 8. 性能策略

### 8.1 大任务

- 分页采集。
- 分批写入数据库。
- 大体积原始响应写文件。
- 任务阶段性提交。
- UI 只读取摘要和分页数据。

### 8.2 数据库

- 使用 WAL。
- 关键查询建索引。
- 大表分页。
- 定期执行优化。

### 8.3 外部请求

- 超时。
- 重试。
- 限流退避。
- 并发上限。
- 成本上限。

## 9. Tauri 权限边界

Tauri 后端必须最小化暴露命令：

- 前端只能调用声明过的 commands。
- 文件访问限制在工作区目录和用户选择的导出路径。
- 网络请求集中在后端请求层。
- 移动端平台能力通过插件封装，不在前端直接访问。

## 10. 后端测试

### 10.1 单元测试

- Provider Registry。
- 参数白名单。
- TikHub 请求构造。
- 模型适配层错误分类。
- Prompt Service 版本选择。
- Schema 校验。
- 字段级证据生成。
- 日志脱敏。

### 10.2 集成测试

- TikHub 连接测试。
- 模型供应商连接测试。
- 自定义 OpenAI-compatible endpoint 连接测试。
- 自然语言采集计划生成。
- AI 整理模板运行。
- Excel 导出。
- PDF 导出。
- Webhook 发送。

### 10.3 回归测试

- 提示词基础样例。
- 非 JSON 输出处理。
- 缺失证据处理。
- 部分成功处理。
- 取消任务后不再发起外部请求。

## 11. 后端验收标准

- TikHub 和至少一个模型供应商连接测试可用。
- 自定义 OpenAI-compatible endpoint 可配置并测试。
- 自然语言采集计划必须经过 Schema 和白名单校验。
- 用户确认前不发起正式 TikHub 采集。
- AI 运行记录包含 Runtime Snapshot。
- 核心 AI 字段包含来源证据。
- 导出前完整性检查能阻止无证据核心洞察。
- 日志、导出、Webhook、备份不泄露 API Key。
- 任务失败时记录失败阶段、原因、可重试性和下一步。
