# 智能数据整理平台技术框架总览

版本：v0.4
日期：2026-07-17
关联 PRD：`智能数据整理平台_PRD.md` v0.7
适用范围：macOS 单端 MVP  
文档定位：技术架构主索引，定义前端、后端、本地数据库、私有 API 配置注册表、外部 API、模型供应商、导出与安全边界

> 决策覆盖：从 v0.4 起，用户确认的统一 JSON API 配置方案覆盖本文及旧文档中“API Key 必须进入 macOS Keychain／系统安全存储”的既有口径。旧 Keychain 条目只用于旧版本回滚，不再由新版本读取或删除。

## 1. 总体目标

MVP 要做的是本地优先的 macOS 智能数据整理平台，而不是云端爬虫平台、团队协作平台或 AI 聊天应用。

核心目标：

- 用户在本机配置 TikHub 和模型供应商密钥。
- 用户通过表单或自然语言创建采集任务。
- 应用通过 TikHub REST API 采集 TikTok、抖音、小红书三个平台公开数据。
- 当前自然语言采集计划由 `local-rule-engine/rule-parser-v1` 生成，不调用 AI API。
- 本次 AI API 范围只包含保存、完整性校验、查看和当前配置切换；真实模型调用、AI 整理和报告洞察仍是后续能力。
- 工作区、任务、原始数据、标准化数据、AI 运行记录、提示词版本、报告和导出文件默认都保存在本机本地。
- MVP 必须支持提示词优化、提示词版本管理、Schema 校验、证据引用和基础回归样例。
- 对外导出以 Excel 工作簿和 PDF 报告为主。

## 2. 架构原则

### 2.1 本地优先

- 不设置自有云端 API 代理。
- 不接远端数据库。
- 不做云端任务、云端长期队列或云端密钥托管。
- TikHub 与 AI API 配置以当前工作区 `secrets/api-config.json` 为唯一主数据源；API Key 以明文凭据保存在该私有 JSON 中。
- 私有 JSON 所在目录固定为 `0700`，文件固定为 `0600`。该权限可限制其他本机用户，但不能防止同一 macOS 账户下的恶意进程读取。
- SQLite 只保存可重建镜像、密钥引用元数据、审计记录和不含明文密钥的运行快照。
- 只有实际调用第三方服务时，本机客户端才通过 HTTPS 把对应密钥发送给用户选择的服务；当前 AI 配置校验不发起真实模型请求。
- 不上传原始社交数据到自有服务器或远端数据库。
- 不上传 AI 输出和报告文件到自有服务器或远端数据库。
- 工作区、任务队列、缓存、数据库、AI 运行记录、报告和导出文件都在用户设备本地。

### 2.2 macOS 优先，后续多端独立本地

- MVP 只做 macOS 桌面端完整工作流，包括配置、采集、整理、预览、导出、Webhook。
- Windows、iOS、Android 后续扩展时，默认都是各自独立的本地工作区，不默认跨端同步。
- 账号功能、远端数据库、远端同步、云端任务、云端密钥托管和跨设备连续工作流不进入 MVP，后续作为独立版本重新评审。
- 同一套业务模型、同一套提示词版本、同一套 Schema 校验规则。
- 后续移动端 UI 不照搬桌面复杂表格，使用记录卡片、折叠字段和轻量审核流。

### 2.3 AI 受控

- 当前自然语言采集由本地规则引擎输出结构化采集计划。
- 应用负责参数白名单校验、成本估算、用户确认和 TikHub REST API 调用。
- 当前 AI 配置只做保存、完整性校验、查看和切换，不进入任务执行链路。
- 未来接入真实模型时，AI prompt 不得包含 TikHub API Key 或模型供应商 API Key。
- MVP 不让 AI 通过 MCP 或工具链直接调用 TikHub。
- TikHub MCP 放到 V1 高级 Agent 模式。

### 2.4 反数据 slop

数据 slop 指看起来已经整理完成，但缺少来源引用、字段级证据、Schema 校验、冲突说明、置信度、人工确认或失败边界的数据结果。

技术层必须保证：

- 每条核心 AI 洞察能追溯到原始记录 ID。
- 每次 AI 运行保存供应商、模型、协议格式、提示词版本、Schema 版本和运行快照。
- 失败、部分成功、待人工确认必须显式保存和展示。
- Excel 和 PDF 不输出未校验且无来源引用的核心结论。

## 3. 推荐技术栈

### 3.1 客户端框架

- Tauri v2：macOS 桌面壳，后续可扩展到 Windows、iOS、Android。
- React：前端 UI。
- TypeScript：类型约束和大型前端工程维护。
- Vite：开发构建。
- Radix UI 或同类无样式基础组件：弹窗、菜单、选择器、Tabs、Tooltip、Popover。
- TanStack Query：任务、数据、连接测试和异步状态管理。
- TanStack Table：桌面端复杂表格。

当前界面使用普通 CSS 和 React 局部状态。只有出现经过验收的跨页面共享状态或图表需求后，才评估全局状态库和图表库。

### 3.2 本地后端

- Tauri Rust Commands：本地后端入口，承接密钥、数据库、文件、任务队列、外部请求和系统能力。
- Rust HTTP Client：统一调用 TikHub 和模型供应商。
- Rust 任务队列：管理采集、AI 整理、导出、Webhook 等本地任务。
- 私有 API 配置注册表：在当前工作区 `secrets/api-config.json` 统一保存 TikHub、AI 配置和明文凭据，使用原子写入、进程内互斥和严格权限校验。
- 本地文件系统：保存原始响应、导出文件、备份和临时文件。

### 3.3 数据库与存储

- SQLite：本地结构化数据库。
- WAL 模式：提高本地读写并发和任务运行期间的响应性。
- 外键约束：保证任务、记录、AI 运行、报告之间的关系完整。
- 文件存储：大体积原始响应、Excel 文件、PDF 文件、备份包不直接塞入 SQLite。
- 明文密钥不进入 SQLite；SQLite 中的配置、`secret_ref` 和当前连接器状态均为可从 JSON 重建的派生镜像。

### 3.4 导出

- Rust 后端负责报告模型校验、Excel 与 PDF 生成、文件落盘和完整性检查。
- Excel 使用 `rust_xlsxwriter` 生成真实单元格、多工作表和元数据。
- PDF 由 Rust 导出器生成，前端只触发任务、展示进度和结果路径。
- MVP 不保留第二套前端导出实现；只有出现浏览器端独立导出需求时才重新评估 JavaScript 导出库。

### 3.5 外部服务

- TikHub REST API：MVP 核心数据采集路径。
- 模型供应商配置：支持 OpenAI、Anthropic、Gemini、Ollama、自定义 OpenAI-compatible endpoint；本次只做本地完整性校验，不调用模型 API。
- Webhook：任务完成摘要发送到用户配置地址，n8n 通过该路径轻集成。
- TikHub MCP：V1 高级 Agent 模式，不进入 MVP 核心路径。

## 4. 分层架构


## 5. 核心模块

### 5.1 工作区模块

职责：

- 创建本地工作区。
- 管理项目、任务、报告、设置。
- 展示本地存储占用。
- 手动备份和恢复。

边界：

- 不做云同步。
- 不做账号体系。
- 不接远端数据库。
- 不做远端同步。
- 不做云端任务。
- 不做云端密钥托管。
- 不做团队工作区。
- 不把 `secrets/api-config.json` 或任何明文密钥写入备份包。

### 5.2 密钥与供应商模块

职责：

- 在统一 JSON 中保存多条命名 TikHub 配置和多条命名 AI 配置。
- 保存每类唯一的当前配置指针，允许同一供应商存在多个账号或端点。
- 将明文凭据与配置对象分开保存，并向前端只返回首尾脱敏视图。
- 将安全配置视图确定性投影到 SQLite 派生镜像。
- 支持自定义 OpenAI-compatible endpoint。
- TikHub 执行真实账户连通测试；AI 只校验协议、地址、模型和密钥完整性。
- 密钥脱敏显示。

核心实体：

- `api_profile_registry`
- `active_profile_ids`
- `tikhub_profile`
- `ai_profile`
- `credential`
- SQLite 派生 `secret_ref`、`tikhub_connector` 和 `model_provider`

### 5.3 TikHub 采集模块

职责：

- 平台和数据类型能力注册。
- 表单式参数校验。
- 自然语言采集计划校验。
- 成本估算。
- TikHub REST API 调用。
- 原始响应摘要保存。
- 失败重试和限流处理。

首批平台：

- TikTok
- 抖音
- 小红书

### 5.4 AI 与提示词模块

当前实现边界：AI API 当前配置不参与任务执行，自然语言计划仍由本地规则引擎生成。以下真实模型执行、提示词运行和证据生成职责在接入模型执行链路时启用。

职责：

- 维护提示词模板。
- 维护提示词版本。
- 管理基础回归样例。
- 生成自然语言采集计划。
- 执行 AI 整理模板。
- 结构化输出 Schema 校验。
- 记录 Runtime Snapshot。
- 生成字段级证据。

MVP 提示词优化范围：

- 采集计划提示词。
- 缺失条件追问提示词。
- 结构化输出提示词。
- 证据引用提示词。
- 失败边界提示词。
- 成本控制提示词。
- 多供应商兼容提示词。

### 5.5 任务队列模块

职责：

- 创建任务。
- 排队、取消、重试。
- 分阶段保存任务结果。
- 记录运行日志。
- 记录成本。
- 支持部分成功。

任务状态：

- 草稿
- 等待确认
- 排队中
- 运行中
- 部分成功
- 成功
- 失败
- 已取消
- 待人工确认

### 5.6 数据查看模块

职责：

- 原始数据视图。
- 标准化数据视图。
- AI 结果视图。
- 来源追溯视图。
- 运行日志视图。
- 桌面端复杂表格。

### 5.7 导出模块

职责：

- 生成 Excel 工作簿。
- 生成 PDF 报告。
- 导出前完整性检查。
- 导出日志。
- 本地路径记录。

导出门禁：

- 核心洞察有证据引用。
- PDF 字体可用。
- Excel 工作表完整。
- 不包含 API Key。
- 部分成功和待确认状态不被隐藏。

### 5.8 Webhook 与 n8n 轻集成模块

职责：

- 用户配置 Webhook URL。
- 任务完成后发送摘要。
- 记录成功和失败。
- 不保证离线补发。

边界：

- 不把 n8n 放在 MVP 核心导航。
- 不做复杂工作流编排。
- V1 再提供 n8n 模板。

## 6. 两种核心调用流程

### 6.1 表单式精确采集

```text
用户选择平台、数据类型、国家 / 地区和筛选条件
  -> 前端校验表单
  -> 后端校验参数白名单
  -> 估算 TikHub 请求量和成本
  -> 用户确认
  -> 后端调用 TikHub REST API
  -> 保存原始响应摘要和标准化记录
  -> 前端预览
  -> 用户保存 / 整理 / 导出
```

### 6.2 自然语言智能采集

```text
用户输入自然语言需求
  -> 本地规则引擎生成结构化采集计划
  -> 应用校验计划 Schema
  -> 应用校验平台、数据类型、国家 / 地区、参数白名单和成本
  -> 前端展示计划，允许用户修改或拒绝
  -> 用户确认
  -> 后端调用 TikHub REST API
  -> 保存采集计划、最终请求摘要、响应摘要和标准化记录
  -> 前端预览
  -> 用户保存 / 整理 / 导出
```

## 7. 统一 API 配置注册表设计

当前工作区 `secrets/api-config.json` 是 TikHub 与 AI API 配置的唯一主数据源。SQLite 只保留可重建镜像和运行所需引用，不得反向覆盖 JSON。

核心字段：

- `schema_version`
- `active_profile_ids.tikhub`
- `active_profile_ids.ai`
- `tikhub_profiles`
- `ai_profiles`
- `credentials`

设计要求：

- 配置与凭据使用稳定 UUID 和独立修订号，配置对象不重复保存密钥。
- TikHub 名称在 TikHub 配置内唯一，AI 名称在 AI 配置内唯一。
- 供应商类型和协议格式分离，支持同一供应商多个账号或端点。
- 支持自定义 OpenAI-compatible endpoint。
- 支持不需要密钥的 Ollama。
- 第一条验证成功的配置自动成为当前配置；后续配置由用户显式切换。
- 未验证、测试失败或需重新绑定的配置不能激活；删除当前配置后当前指针置空，不静默选择其他配置。
- 新鲜排队任务在首次领取时绑定当时的当前 TikHub 配置；运行中和恢复任务继续使用不含明文密钥的不可变 Runtime Snapshot。
- JSON 损坏、版本未知或权限不安全时失败关闭，不覆盖原文件；工作区浏览与历史数据仍可访问。

## 8. 安全边界

### 8.1 密钥边界

- API Key 以明文保存到当前工作区私有 `secrets/api-config.json`，目录权限为 `0700`，文件权限为 `0600`。
- SQLite、日志、导出、Webhook 和应用备份不出现完整 API Key。
- 同一 macOS 账户下的恶意进程仍可能读取 JSON；这是取消 Keychain 后明确接受的安全降级。
- 旧 Keychain 条目不自动读取或删除，避免触发系统权限窗并保留旧版本回滚能力。
- 首次导入旧 SQLite 元数据时，配置统一标记为 `needs_rebind`，当前指针置空；用户必须在对应管理弹窗重新输入一次 TikHub Token 或 AI API Key。
- AI prompt 不包含 TikHub API Key 或模型供应商 API Key。

### 8.2 请求边界

- 所有外部请求使用 HTTPS，用户自定义本地网关除外。
- TikHub 请求必须走统一请求层。
- 模型供应商请求必须走模型适配层。
- 自定义 Base URL 必须在 UI 中明确展示并由用户确认。

### 8.3 数据边界

- 默认不上传原始社交数据到自有服务器。
- 默认不上传 AI 输出到自有服务器。
- Webhook 只发送用户启用的摘要，不发送密钥。
- 崩溃日志和诊断日志必须脱敏。

## 10. MVP 验收总表

| 领域 | 必须通过 |
| --- | --- |
| macOS 壳 | macOS 能启动并进入本地工作区 |
| API 配置 | TikHub 与 AI 均可保存多条配置、查看脱敏列表并切换当前配置；TikHub 真实测试，AI 只做完整性校验 |
| 密钥 | 明文只出现在权限正确的工作区私有 JSON，不出现在 SQLite、日志、导出、Webhook 或备份 |
| 表单采集 | TikTok、抖音、小红书三平台各完成至少一个采集任务 |
| 自然语言采集 | 本地规则引擎生成可编辑采集计划，确认后由应用调用 TikHub REST API |
| 提示词优化 | 固定样例能验证采集计划 Schema、整理结果 Schema、证据引用和失败边界 |
| 数据查看 | 原始数据、标准化数据、AI 结果和来源追溯可查看 |
| 导出 | Excel 多工作表和 PDF 报告可生成，中文正常，不含密钥 |
| Webhook | 成功和失败都有日志，不发送 API Key |
| 反数据 slop | 核心洞察都能追溯到原始记录和模型运行记录 |

## 11. 风险与优化方向

### 11.1 风险

- 后续扩展多端时，Tauri 移动端插件能力与桌面端存在差异。
- 后续扩展到 iOS 和 Android 时，后台任务限制会影响长任务。
- 账号、远端数据库、远端同步、云端任务和云端密钥托管不进入 MVP，后续引入时会改变安全、隐私和数据一致性边界。
- 自定义 OpenAI-compatible endpoint 协议细节可能不一致。
- 工作区私有 JSON 包含明文密钥，无法防止同一 macOS 账户下的恶意进程读取。
- 旧配置迁移后必须重新输入密钥一次；旧 Keychain 条目需由用户确认新版本正常后手动清理。
- PDF 中文字体、分页和表格可能需要反复校验。
- 大任务会导致本地数据库、文件和内存压力。

### 11.2 优化方向

- 长任务优先建议桌面端执行。
- 移动端减少复杂表格，提供审核和查看体验。
- Provider Registry 增加能力标签和连接测试。
- 导出前执行完整性检查。
- SQLite 使用 WAL、索引、分页查询和归档策略。
