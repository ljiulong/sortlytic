# 智能数据整理平台技术框架总览

版本：v0.2  
日期：2026-07-05  
关联 PRD：`智能数据整理平台_PRD.md` v0.6  
适用范围：macOS 单端 MVP  
文档定位：技术架构主索引，定义前端、后端、本地数据库、外部 API、模型供应商、导出与安全边界

## 1. 总体目标

MVP 要做的是本地优先的 macOS 智能数据整理平台，而不是云端爬虫平台、团队协作平台或 AI 聊天应用。

核心目标：

- 用户在本机配置 TikHub 和模型供应商密钥。
- 用户通过表单或自然语言创建采集任务。
- 应用通过 TikHub REST API 采集 TikTok、抖音、小红书三个平台公开数据。
- AI 只生成采集计划、结构化整理结果和报告洞察，不直接持有 TikHub API Key。
- 工作区、任务、原始数据、标准化数据、AI 运行记录、提示词版本、报告和导出文件默认都保存在本机本地。
- MVP 必须支持提示词优化、提示词版本管理、Schema 校验、证据引用和基础回归样例。
- 对外导出以 Excel 工作簿和 PDF 报告为主。

## 2. 架构原则

### 2.1 本地优先

- 不设置自有云端 API 代理。
- 不接远端数据库。
- 不做云端任务、云端长期队列或云端密钥托管。
- API Key 只在本地持久化保存，调用第三方服务时由本机客户端直接通过 HTTPS 发送给对应服务。
- 不上传原始社交数据到自有服务器或远端数据库。
- 不上传 AI 输出和报告文件到自有服务器或远端数据库。
- 工作区、任务队列、缓存、数据库、AI 运行记录、报告和导出文件都在用户设备本地。

### 2.2 macOS 优先，后续多端独立本地

- MVP 只做 macOS 桌面端完整工作流，包括配置、采集、整理、预览、导出、Webhook。
- Windows、iOS、Android 后续扩展时，默认都是各自独立的本地工作区，不默认跨端同步。
- 账号功能、远端数据库、远端同步、云端任务、云端密钥托管和跨设备连续工作流不进入 MVP，后续作为独立版本重新评审。
- 同一套业务模型、同一套提示词版本、同一套 Schema 校验规则。
- 后续移动端 UI 不照搬桌面复杂表格，使用记录卡片、折叠字段和轻量审核流。

### 2.3 AI 受控

- 自然语言采集中，AI 只输出结构化采集计划。
- 应用负责参数白名单校验、成本估算、用户确认和 TikHub REST API 调用。
- AI 不直接持有 TikHub API Key。
- MVP 不让 AI 通过 MCP 或工具链直接调用 TikHub。
- TikHub MCP 放到 V1 高级 Agent 模式。

### 2.4 反数据 slop

数据 slop 指看起来已经整理完成，但缺少来源引用、字段级证据、Schema 校验、冲突说明、置信度、人工确认或失败边界的数据结果。

技术层必须保证：

- 每条核心 AI 洞察能追溯到原始记录 ID。
- 每次 AI 运行保存供应商、模型、协议格式、提示词版本、Schema 版本和运行快照。
- 失败、部分成功、待人工确认必须显式保存和展示。
- Excel 和 PDF 不输出未校验且无来源引用的核心结论。

## 3. 推荐技术栈

### 3.1 客户端框架

- Tauri v2：macOS 桌面壳，后续可扩展到 Windows、iOS、Android。
- React：前端 UI。
- TypeScript：类型约束和大型前端工程维护。
- Vite：开发构建。
- Tailwind CSS：设计系统和响应式样式。
- Radix UI 或同类无样式基础组件：弹窗、菜单、选择器、Tabs、Tooltip、Popover。
- TanStack Query：任务、数据、连接测试和异步状态管理。
- Zustand：本地 UI 状态，例如侧栏、筛选、当前视图、临时草稿。
- TanStack Table：桌面端复杂表格。
- ECharts：概览图表和报告图表预览。

### 3.2 本地后端

- Tauri Rust Commands：本地后端入口，承接密钥、数据库、文件、任务队列、外部请求和系统能力。
- Rust HTTP Client：统一调用 TikHub 和模型供应商。
- Rust 任务队列：管理采集、AI 整理、导出、Webhook 等本地任务。
- 系统安全存储：MVP 使用 macOS 钥匙串；Windows 凭据管理器、iOS Keychain、Android Keystore 放到后续多端扩展。
- 本地文件系统：保存原始响应、导出文件、备份和临时文件。

### 3.3 数据库与存储

- SQLite：本地结构化数据库。
- WAL 模式：提高本地读写并发和任务运行期间的响应性。
- 外键约束：保证任务、记录、AI 运行、报告之间的关系完整。
- 文件存储：大体积原始响应、Excel 文件、PDF 文件、备份包不直接塞入 SQLite。
- 密钥不进入 SQLite，只保存 `secret_ref`。

### 3.4 导出

- ExcelJS：生成 `.xlsx` 多工作表工作簿。
- `@react-pdf/renderer`：生成 PDF 报告。
- `pdf-lib`：PDF 元数据、合并、后处理和附录页。

### 3.5 外部服务

- TikHub REST API：MVP 核心数据采集路径。
- 模型供应商 API：OpenAI、Gemini、Claude、DeepSeek、Qwen、Moonshot、OpenRouter、Ollama、自定义 OpenAI-compatible endpoint。
- Webhook：任务完成摘要发送到用户配置地址，n8n 通过该路径轻集成。
- TikHub MCP：V1 高级 Agent 模式，不进入 MVP 核心路径。

## 4. 分层架构

```text
前端 UI 层
  - 页面、组件、表格、图表、表单、提示词模板编辑、任务预览
  |
  v
前端应用服务层
  - 状态管理、表单 Schema、查询缓存、Tauri invoke 封装
  |
  v
Tauri 本地后端层
  - 命令入口、权限边界、任务队列、外部请求、导出、文件读写
  |
  v
领域服务层
  - TikHub 采集、模型供应商适配、提示词运行、Schema 校验、报告生成
  |
  v
本地数据层
  - SQLite、文件存储、系统安全存储
  |
  v
外部服务
  - TikHub REST API、模型供应商 API、Webhook 目标
```

## 5. 核心模块

### 5.1 工作区模块

职责：

- 创建本地工作区。
- 管理项目、任务、报告、设置。
- 展示本地存储占用。
- 手动备份和恢复。

边界：

- 不做云同步。
- 不做账号体系。
- 不接远端数据库。
- 不做远端同步。
- 不做云端任务。
- 不做云端密钥托管。
- 不做团队工作区。
- 不把密钥写入备份包。

### 5.2 密钥与供应商模块

职责：

- 保存 TikHub API Key。
- 保存模型供应商密钥。
- 支持 Provider Registry。
- 支持自定义 OpenAI-compatible endpoint。
- 连接测试。
- 密钥脱敏显示。

核心实体：

- `secret_ref`
- `model_provider`
- `model_profile`
- `provider_health_check`

### 5.3 TikHub 采集模块

职责：

- 平台和数据类型能力注册。
- 表单式参数校验。
- 自然语言采集计划校验。
- 成本估算。
- TikHub REST API 调用。
- 原始响应摘要保存。
- 失败重试和限流处理。

首批平台：

- TikTok
- 抖音
- 小红书

### 5.4 AI 与提示词模块

职责：

- 维护提示词模板。
- 维护提示词版本。
- 管理基础回归样例。
- 生成自然语言采集计划。
- 执行 AI 整理模板。
- 结构化输出 Schema 校验。
- 记录 Runtime Snapshot。
- 生成字段级证据。

MVP 提示词优化范围：

- 采集计划提示词。
- 缺失条件追问提示词。
- 结构化输出提示词。
- 证据引用提示词。
- 失败边界提示词。
- 成本控制提示词。
- 多供应商兼容提示词。

### 5.5 任务队列模块

职责：

- 创建任务。
- 排队、暂停、取消、重试。
- 分阶段保存任务结果。
- 记录运行日志。
- 记录成本。
- 支持部分成功。

任务状态：

- 草稿
- 等待确认
- 排队中
- 运行中
- 部分成功
- 成功
- 失败
- 已取消
- 待人工确认

### 5.6 数据查看模块

职责：

- 原始数据视图。
- 标准化数据视图。
- AI 结果视图。
- 来源追溯视图。
- 运行日志视图。
- 桌面端复杂表格。
- 移动端记录卡片。

### 5.7 导出模块

职责：

- 生成 Excel 工作簿。
- 生成 PDF 报告。
- 导出前完整性检查。
- 导出日志。
- 本地路径记录。

导出门禁：

- 核心洞察有证据引用。
- PDF 字体可用。
- Excel 工作表完整。
- 不包含 API Key。
- 部分成功和待确认状态不被隐藏。

### 5.8 Webhook 与 n8n 轻集成模块

职责：

- 用户配置 Webhook URL。
- 任务完成后发送摘要。
- 记录成功和失败。
- 不保证离线补发。

边界：

- 不把 n8n 放在 MVP 核心导航。
- 不做复杂工作流编排。
- V1 再提供 n8n 模板。

## 6. 两种核心调用流程

### 6.1 表单式精确采集

```text
用户选择平台、数据类型、国家 / 地区和筛选条件
  -> 前端校验表单
  -> 后端校验参数白名单
  -> 估算 TikHub 请求量和成本
  -> 用户确认
  -> 后端调用 TikHub REST API
  -> 保存原始响应摘要和标准化记录
  -> 前端预览
  -> 用户保存 / 整理 / 导出
```

### 6.2 自然语言智能采集

```text
用户输入自然语言需求
  -> AI 生成结构化采集计划
  -> 应用校验计划 Schema
  -> 应用校验平台、数据类型、国家 / 地区、参数白名单和成本
  -> 前端展示计划，允许用户修改或拒绝
  -> 用户确认
  -> 后端调用 TikHub REST API
  -> 保存采集计划、最终请求摘要、响应摘要和标准化记录
  -> 前端预览
  -> 用户保存 / 整理 / 导出
```

## 7. Provider Registry 设计

Provider Registry 是模型供应商配置的单一事实源。

核心字段：

- `provider_id`
- `display_name`
- `enabled`
- `auth_type`
- `secret_ref`
- `base_url`
- `api_format`
- `models`
- `default_model`
- `capabilities`
- `region`
- `cost_policy`
- `rate_limit_policy`
- `health_check`

设计要求：

- 不把 OpenAI、Gemini、Claude 写死到业务逻辑里。
- 供应商和协议格式分离。
- 模型唯一标识使用 `provider_id + model_id`。
- 支持自定义 OpenAI-compatible endpoint。
- 支持本地模型网关。
- 每次运行保存 Runtime Snapshot，保证历史任务可复现。

## 8. 安全边界

### 8.1 密钥边界

- API Key 只进系统安全存储。
- SQLite 只保存 `secret_ref`。
- 日志、导出、Webhook 不出现完整 API Key。
- AI prompt 不包含 TikHub API Key 或模型供应商 API Key。

### 8.2 请求边界

- 所有外部请求使用 HTTPS，用户自定义本地网关除外。
- TikHub 请求必须走统一请求层。
- 模型供应商请求必须走模型适配层。
- 自定义 Base URL 必须在 UI 中明确展示并由用户确认。

### 8.3 数据边界

- 默认不上传原始社交数据到自有服务器。
- 默认不上传 AI 输出到自有服务器。
- Webhook 只发送用户启用的摘要，不发送密钥。
- 崩溃日志和诊断日志必须脱敏。

| 领域 | 必须通过 |

| macOS 壳 | macOS 能启动并进入本地工作区 |
| 密钥 | TikHub 和至少一个模型供应商连接测试通过，日志不泄露密钥 |
| 表单采集 | TikTok、抖音、小红书三平台各完成至少一个采集任务 |

## 11. 风险与优化方向

### 11.1 风险

- 后续扩展多端时，Tauri 移动端插件能力与桌面端存在差异。
- 后续扩展到 iOS 和 Android 时，后台任务限制会影响长任务。
- 账号、远端数据库、远端同步、云端任务和云端密钥托管不进入 MVP，后续引入时会改变安全、隐私和数据一致性边界。
- 自定义 OpenAI-compatible endpoint 协议细节可能不一致。
- PDF 中文字体、分页和表格可能需要反复校验。
- 大任务会导致本地数据库、文件和内存压力。

### 11.2 优化方向

- 长任务优先建议桌面端执行。
- 移动端减少复杂表格，提供审核和查看体验。
- Provider Registry 增加能力标签和连接测试。
- 导出前执行完整性检查。
- SQLite 使用 WAL、索引、分页查询和归档策略。
