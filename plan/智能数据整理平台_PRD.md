# 智能数据整理平台 PRD

版本：v0.8
日期：2026-07-16
文档类型：产品需求文档  
目标平台：MVP 先做 macOS 桌面端  
产品方向：本地优先、无账号体系、无远端数据库、无远端同步、无云端任务、无云端密钥托管、单用户本地工作区、TikTok / 抖音 / 小红书三平台采集、多供应商模型智能整理、Excel 表格与 PDF 报告导出、n8n 轻集成

## 1. 背景与定位

### 1.1 产品背景

内容、品牌、投研、市场和 AI 数据团队越来越依赖公开社交平台数据做趋势判断、竞品监控、用户洞察和训练数据整理。现有工具常见问题包括：

- 数据采集和 AI 分析割裂，用户需要在多个工具之间手动搬运数据。
- 原始数据、AI 结论和来源证据之间缺少可追溯关系。
- 多模型能力分散，不同模型供应商、协议格式、成本、提示词和输出版本难以统一管理。
- 社媒数据采集存在费用、合规、限流和平台覆盖问题，不适合自研爬虫作为第一选择。
- 导出能力不足，尤其缺少可交付给业务方、客户或研究对象的 Excel 表格和 PDF 报告。

TikHub 提供跨社交平台的数据 API、MCP 工具和数据集能力。产品 MVP 将先基于 TikHub REST API，在 macOS 上构建一个本地优先的智能数据整理工作台；MCP、更多平台、更多端和账号能力后续再扩展。

### 1.2 一句话定位

一个面向个人研究者、内容运营、品牌营销和 AI 数据整理场景的本地优先数据工作台。MVP 先在 macOS 上支持从 TikHub 采集 TikTok、抖音、小红书公开数据，并用多供应商模型完成结构化整理、洞察生成、Excel 表格导出和 PDF 报告导出。

### 1.3 产品原则

- 本地优先：工作区、任务、原始数据、AI 运行记录、报告和导出文件默认只在用户本机持久化保存。
- 可追溯：每条 AI 洞察必须能回到原始数据、采集任务、模型、提示词版本和生成时间。
- 可控成本：所有正式采集运行前必须读取 TikHub 实时报价、免费额度与充值余额，并显示请求量、报价上限和可用额度合计。
- 不做黑盒：AI 可以辅助整理，但不能覆盖原始数据，也不能隐藏失败原因。
- 不自研爬虫：MVP 只通过 TikHub 官方能力获取数据，不绕过平台限制。
- 双入口采集：同时支持表单式精确采集和自然语言智能采集。
- 受控 AI 调用：自然语言模式下，AI 只生成结构化采集计划，应用负责白名单校验、成本确认和 TikHub API 调用，AI 不能直接接触明文密钥。
- macOS 优先：MVP 只做 macOS 桌面端完整工作流；Windows、iOS、Android 后续扩展为各自独立的本地工作区，不默认跨端同步。
- 无账号无远端：MVP 不做账号体系、远端数据库、远端同步、多设备自动同步、云端任务或云端密钥托管。

## 2. 目标与非目标

### 2.1 MVP 目标

- 支持 TikTok、抖音、小红书三个平台的数据采集。
- 支持 TikHub API Key 本地保存与连接测试。
- 支持 OpenAI、Gemini、Claude 作为内置模型供应商，并支持 DeepSeek、Qwen、Moonshot、OpenRouter、Ollama、自定义 OpenAI-compatible 接口等扩展供应商的本地配置与连接测试。
- 支持单用户本地工作区，不做团队协作。
- 支持两种采集入口：表单式精确采集、自然语言智能采集。
- 支持任务创建、运行、取消、失败重试、日志查看、成本估算。
- 支持原始数据、清洗数据、AI 结构化结果并排查看。
- 支持摘要、情绪分析、主题分类、实体抽取、评论聚类、爆点拆解。
- MVP 必须内置提示词优化能力，不能只停留在 API Key 导入和模型调用；提示词优化要覆盖自然语言采集计划生成、AI 整理模板、结构化输出、证据引用和失败边界。
- 支持 Excel 表格（.xlsx）和 PDF 报告导出。
- 支持 Webhook 作为 n8n 等自动化工具的接入路径，但不把 n8n 作为核心卖点。
- 支持 macOS 桌面端运行。

### 2.2 成功指标

- 用户首次完成 TikHub 连接配置小于 3 分钟。
- 用户首次创建并完成一个采集与 AI 整理任务小于 8 分钟。
- macOS 端采集任务成功率大于 95%，失败任务必须给出可执行错误说明。
- AI 结构化字段在人工抽检中准确率达到 85% 以上。
- 100% 的 AI 洞察可以追溯到原始记录和模型运行记录。
- 100% 的导出报告不包含明文 API Key。
- PDF 报告在 macOS 默认预览或阅读器中中文显示正常。

### 2.3 非目标范围

MVP 不做以下能力：

- 不做团队工作区、成员邀请、角色权限、审批流。
- 不做账号体系、登录、注册和远端用户身份。
- 不接远端数据库。
- 不做云端密钥托管。
- 不做云端任务或云端长期任务执行。
- 不做多设备自动同步。
- 不做自研爬虫。
- 不做非公开数据采集。
- 不做自动评论、自动私信、自动发布等平台交互动作。
- 不做完整 BI 平台。
- 不做自有大模型训练。
- 不做直播 WebSocket 数据接入。
- 不做高级向量知识库和训练集管理。

## 3. 用户与场景

### 3.1 目标用户

#### 内容研究员

关注热门内容、爆款结构、创作者行为和内容趋势，需要快速从多个平台采集样本并生成洞察。

#### 品牌与营销人员

关注品牌提及、竞品内容、达人表现、用户反馈和评论情绪，需要定期生成可分享报告。

#### 投研与情报分析人员

关注公开社交信号、事件扩散路径、情绪变化和风险提示，需要保留来源证据和审计记录。

#### AI 数据整理人员

需要将公开社媒数据清洗为 Excel 表格和可追溯报告，用于数据分析、业务交付、RAG 或训练前筛选。

### 3.2 核心场景

#### 场景一：关键词趋势研究

用户输入关键词、平台、时间范围，系统采集相关内容，去重后用 AI 生成趋势摘要、主题分类和高频观点。

#### 场景二：社媒账号内容分析

用户输入一个或多个社媒账号链接，系统采集账号公开内容，分析内容主题、互动表现、评论情绪和爆款因素。

#### 场景三：竞品监控

用户创建竞品关键词或社媒账号任务，定期手动运行或通过导出接入自动化系统，生成竞品变化报告。

#### 场景四：评论洞察

用户采集指定帖子、视频或笔记的评论，系统进行情绪识别、问题聚类、购买意图识别和代表性评论引用。

#### 场景五：报告交付

用户将整理结果导出为 Excel 表格和 PDF 报告，交给业务方、客户或团队成员阅读。Excel 承载明细数据，PDF 承载结论、证据、数据概览和免责声明。

## 4. 关键决策

### 4.1 MVP 首批平台

MVP 确认支持以下三个平台：

- TikTok
- 抖音
- 小红书

选择理由：

- 覆盖国际短视频、国内短视频和中文种草社区。
- 适合内容研究、营销、品牌监控和评论洞察。
- 平台数量足够验证采集、标准化、AI 整理、证据追溯和报告导出的完整链路。
- YouTube、Instagram 放到后续版本，避免 MVP 被平台适配拖散。

### 4.2 单用户本地工作区

MVP 只做 macOS 本机上的单用户本地工作区，不做账号体系，也不接远端数据库。

保留能力：

- 本地项目。
- 本地任务。
- 本地数据资产。
- 本地报告。
- 本地密钥。

不做能力：

- 登录和注册。
- 远端数据库。
- 远端同步。
- 多设备自动同步。
- 团队成员。
- 角色权限。
- 云端共享。
- 多人协作审稿。

### 4.3 密钥仅本地持久化

TikHub 和用户配置的所有模型供应商 API Key 只在用户本地设备持久化保存，不上传自有服务器或远端数据库。运行任务时，本机客户端会通过 HTTPS 把对应密钥发送给用户选择的第三方服务，例如 TikHub 或模型供应商。

产品承诺：

- 默认不把 API Key 发送给自有服务器、远端数据库或无关第三方。
- 默认不把原始数据发送给自有服务器或远端数据库。
- 日志、报告、导出文件中不得出现完整 API Key。
- 用户换设备后需要重新配置 API Key。
- 若未来支持账号、远端数据库、云端同步、云端任务或云端密钥托管，必须作为独立版本重新评审。

### 4.4 PDF 是 MVP 必需能力

MVP 必须支持 PDF 报告导出，不能后置。

PDF 用途：

- 对外交付。
- 内部汇报。
- 研究归档。
- 客户沟通。
- 任务成果沉淀。

### 4.5 n8n 不做核心卖点，但不能忽略

MVP 不把 n8n 放在主导航或核心转化路径中，但必须提供接入路径：

- 支持导出 Excel 表格（.xlsx）。
- 支持任务完成后发送 Webhook。
- 在帮助文档中说明如何通过 Webhook 或文件导出接入 n8n。

V1 再考虑提供 n8n 模板。

## 5. 用户流程

### 5.1 首次使用流程

1. 用户打开应用。
2. 创建本地工作区。
3. 配置 TikHub API Key。
4. 配置至少一个模型供应商 API Key。
5. 运行连接测试。
6. 创建第一个采集任务。
7. 查看采集结果。
8. 执行 AI 整理。
9. 生成报告。
10. 导出 Excel 表格、PDF 报告或发送 Webhook 摘要。

### 5.2 双入口采集流程

MVP 必须同时支持两种采集方式。

#### 方式一：表单式精确采集

适用场景：

- 用户知道要采集哪个平台。
- 用户知道要采集的数据类型。
- 用户希望明确选择国家/地区条件。
- 用户希望强控制成本和参数。

流程：

```text
用户选择平台
  |
  v
用户选择数据类型
  |
  v
用户选择国家/地区和其他筛选条件
  |
  v
系统估算 TikHub 请求量和成本
  |
  v
用户确认运行
  |
  v
应用调用 TikHub API
  |
  v
应用内预览采集数据
  |
  v
用户保存到本地工作区
  |
  v
用户导出 Excel 工作簿 / PDF 报告
```

表单式采集要求：

- 平台和目标数据类型必须是显式控件；目标数据类型允许多选。
- 国家/地区使用单一可搜索下拉控件，显示中文名称和两位代码，保存值为代码，不提供重复的名称字段和代码字段。
- 年龄范围和性别均为可选筛选条件。年龄使用 0～130 的闭区间，性别允许多选；两者只使用 TikHub 或公开资料明确返回的值，禁止根据头像、姓名、简介、内容等信息推断。
- 所有将被传给 TikHub API 的参数必须在运行前可见。
- 如果某个平台或接口不支持国家/地区筛选，系统不得向该步骤传入地区参数，并必须提示能力限制；所选步骤全部不支持时自动清空并禁用地区选择器。
- 确认运行只在输入、接口能力、年龄与性别筛选、实时报价、可用额度和预算均有效时启用，按钮旁显示第一条可操作的阻塞原因。
- 用户可以先预览少量数据，再决定是否保存完整任务。

#### 方式二：自然语言智能采集

适用场景：

- 用户知道目标，但不知道应该选哪个平台、接口或字段。
- 用户希望用一句话描述数据需求。
- 用户希望 AI 帮忙拆解采集计划。

流程：

```text
用户用自然语言描述需要的数据
  |
  v
AI 分析需求，抽取平台、数据类型、国家/地区、关键词、账号、时间范围等条件
  |
  v
AI 生成结构化采集计划
  |
  v
应用展示采集计划、预计请求量和成本
  |
  v
用户确认或修改计划
  |
  v
应用按确认后的计划直接调用 TikHub REST API
  |
  v
应用内预览收集到的数据
  |
  v
用户保存到本地工作区
  |
  v
用户导出 Excel 工作簿 / PDF 报告
```

自然语言采集要求：

- AI 必须先生成采集计划，不能直接静默调用 TikHub API。
- 用户确认前，不得发起会产生 TikHub 费用的正式采集请求。
- MVP 中 AI 不直接通过 MCP 或工具链调用 TikHub；AI 只负责需求解析和采集计划生成。
- 应用负责读取本地 TikHub API Key，并按用户确认后的计划直接调用 TikHub REST API。
- 应用必须校验 AI 生成的参数是否在平台和接口白名单内。
- 每次 AI 生成的采集计划必须保存到任务记录中，方便追溯。
- 如果 AI 无法确定平台、数据类型，或任务需要国家/地区但相关条件缺失，必须向用户追问，而不是猜测执行。

### 5.3 标准整理流程

```text
用户输入任务条件
  |
  v
系统估算数据量与成本
  |
  v
用户确认运行
  |
  v
TikHub API 获取公开数据

  |

  |

  v

```

### 5.4 macOS MVP 与后续多端扩展

#### macOS MVP

- 完整采集任务。
- 长任务运行。
- 数据表格编辑。
- PDF 生成。
- 本地备份。
- Webhook 测试。
- 本地工作区、SQLite、文件存储和 macOS 钥匙串。

#### 后续 Windows

- 作为独立本地工作区运行，不默认同步 macOS 数据。
- 承接完整桌面工作流。
- 使用 Windows 凭据管理器和本地文件系统。

#### 后续 iOS

- 作为独立本地工作区运行，不默认同步桌面端数据。
- 适合短任务、查看、审核和轻量导出。
- 不承诺长时间后台采集。

#### 后续 Android

- 作为独立本地工作区运行，不默认同步桌面端数据。
- 适合短任务、查看、审核和轻量导出。
- 不承诺长时间后台采集。

后续多端原则：

- 四端扩展时，每个端默认都是独立本地工作区。
- 不做账号体系前，不提供跨端自动同步。
- 可通过手动备份、手动导入或导出文件分享迁移数据，但这不等于远端同步。
- 账号功能、远端数据库、云端同步、云端任务、云端密钥托管和跨设备连续工作流必须作为后续独立版本重新评审。

## 6. 功能需求

### 6.1 本地工作区

#### 需求

- 用户首次进入应用时创建一个本地工作区。
- 本地工作区包含项目、任务、数据、报告、设置。
- 支持修改工作区名称。
- 支持查看本地数据占用空间。
- 支持导出本地备份。
- 支持清空本地数据。

#### 验收标准

- 断网状态下可以打开已有工作区。
- 退出应用后重新打开，任务和报告仍存在。
- 删除工作区数据前必须二次确认。
- 本地备份不包含明文 API Key。

### 6.2 本地密钥管理

#### 支持密钥

- TikHub API Key。
- 内置模型供应商 API Key：OpenAI、Gemini、Claude。
- 扩展模型供应商 API Key：DeepSeek、Qwen、Moonshot、OpenRouter 等。
- 自定义 OpenAI-compatible 接口的 API Key、Base URL 和模型列表。
- 本地模型网关或 Ollama 如无需 API Key，应记录连接地址和模型能力，不强制填写密钥。

#### 需求

- 支持新增、编辑、删除密钥。
- 支持连接测试。
- 支持查看最近一次测试状态。
- 支持给密钥设置别名。
- 支持默认模型供应商选择。
- 密钥输入框默认隐藏明文。
- 日志和导出中不得出现完整密钥。

#### 本地安全策略

- MVP 使用 macOS 系统钥匙串。
- Windows、iOS、Android 后续扩展时分别使用系统凭据管理器、iOS Keychain、Android Keystore 或对应安全存储。
- 若当前平台安全存储不可用，应用必须阻止保存密钥，并提示用户原因。

#### 验收标准

- 重启应用后密钥仍可用。
- 日志中只显示密钥尾号或掩码。
- 删除密钥后无法继续运行依赖该密钥的任务。
- 连接测试失败时给出供应商、状态码或网络错误原因。

### 6.3 TikHub 数据连接

#### 需求

- 支持 TikHub API Key 连接测试。
- 支持按平台选择采集能力。
- 支持多选目标数据类型，并以关键词入口自动串联搜索、内容详情、账号资料、账号作品与评论步骤。
- 支持国家/地区筛选；若 TikHub 对应接口不支持，必须明确提示。
- 国家/地区通过“中文名称 + 两位代码”的单一下拉字段选择和存储，用于限定目标市场、账号筛选、直播候选校验和报告分组；国家/地区缺失时不得自动猜测默认国家。
- 支持可选年龄范围和性别多选。启用后，年龄未知、性别未知、格式异常或不符合筛选条件的账号不进入最终结果；系统继续分页，直到达到合格账号数、数据耗尽或预算上限。
- 年龄和性别只能来自 TikHub 或公开资料的明确字段，不进行外观、姓名、简介或内容推断。
- 同时读取充值余额、免费额度和可用额度合计；额度任一必要值未知时不得确认运行。
- 在计划生成、确认运行和每次正式请求前读取实时接口报价；价格未知或预算不足时不得发起请求。
- 支持任务级请求超时设置。
- 支持限流提示。
- 支持失败重试。
- 支持将 TikHub 请求次数计入任务成本。
- 支持表单式采集和自然语言智能采集共用同一套 TikHub 请求层。
- 支持 TikHub API 参数白名单，防止 AI 生成未授权或高风险调用。

#### MVP 采集类型

TikTok：

- 关键词搜索。
- 视频详情。
- 用户公开信息。
- 用户作品列表。
- 评论采集。

抖音：

- 关键词搜索。
- 视频详情。
- 用户公开信息。
- 用户作品列表。
- 评论采集。

小红书：

- 关键词搜索。
- 笔记详情。
- 用户公开信息。
- 用户笔记列表。
- 评论采集。

三平台通用：

- 公开互动来源账号发现：仅基于公开评论、公开互动归因字段或公开视频 / 笔记 / 直播间关联账号生成候选账号，不采集非公开粉丝关系、登录后可见数据、私信或隐藏互动列表。
- 账号公开信息入库：作品数为 0 的账号允许入库，并标记为“无公开作品样本”，避免把新账号、清空作品账号或仅直播账号误判为无效数据。
- 按国家代码标记或筛选账号：支持使用国家/地区的标准化国家代码、账号公开资料、内容公开元数据和直播公开元数据对账号进行标记或筛选；筛选依据必须展示国家/地区、来源和置信度，明确筛选不得由 AI 推断结果代替。
- 账号合并与过滤：按“平台 + 平台用户 ID”去重，缺少稳定 ID 时使用规范化账号标识；字段优先级为账号公开信息、内容作者信息、评论作者信息，完成合并后再执行年龄和性别筛选。
- 目标轮询与失败隔离：不同目标类型轮询采集，重复或筛选不合格记录不占最终记录数；单目标失败保留证据并继续其他目标，有合格结果且存在失败时标记为部分成功。
- 直播间状态监测：在 TikHub 支持的公开接口范围内，周期性检查候选直播间或主播账号是否开播；不接直播 WebSocket，不采集非公开直播数据，不绕过平台限制。
- 直播标签发现：支持按公开直播标签发现候选直播间，并通过主播账号国家/地区字段的标准化国家代码、来源和置信度进行校验；无法校验时进入候选池或待人工确认。
- 开播触发动作：候选直播间开播后，可触发本机通知、数据快照和报告生成；触发频率、请求上限、成本上限和 Webhook 行为必须在运行前可见并可关闭。

#### 验收标准

- 三个平台各能完成至少一个采集任务。
- 用户可以通过表单选择平台、数据类型、国家/地区并发起采集。
- 用户可以一次选择多个目标数据类型，系统能正确传递作品 ID 和账号 ID，并且依赖步骤结果不占最终数量。
- 用户可以启用或关闭年龄与性别筛选；边界值、单岁区间、多性别选择、未知值和异常值均按明确规则处理。
- 用户可以基于国家/地区的标准化国家代码标记或筛选账号。
- 作品数为 0 的账号可以保存到本地工作区，并带有明确状态标记。
- 用户可以通过自然语言生成采集计划，并在确认后发起采集。
- 任务结果必须保存原始响应摘要和标准化字段。
- TikHub 限流时任务进入可重试失败状态。
- TikHub API Key 无效时任务不能继续运行。
- 设置页同时显示充值余额、免费额度与可用额度合计。
- 价格、额度或预算门禁失败时确认按钮不可用，并显示第一条可操作原因。
- AI 生成的 TikHub 调用参数必须经过应用校验后才能执行。

### 6.4 AI 模型接入

#### 支持供应商

- 内置官方供应商：OpenAI、Gemini、Claude。
- 内置扩展供应商：DeepSeek、Qwen、Moonshot、OpenRouter、Ollama。
- 可选扩展供应商：智谱、MiniMax、火山方舟、阶跃星辰、小米大模型等。
- 自定义接口：支持 OpenAI-compatible 接口，用户可填写 Base URL、API Key、协议格式和模型列表。

#### 供应商配置原则

- 不把产品架构写死为三家模型平台。
- 使用统一的模型供应商注册表作为单一事实源。
- 供应商和协议格式分离：供应商表示平台或私有 endpoint，协议格式表示 `openai`、`anthropic`、`gemini` 或后续兼容层。
- 模型唯一标识使用 `provider_key + model_id`，避免不同供应商下同名模型冲突。
- 每个供应商必须支持启用 / 停用、连接测试、默认模型、模型列表、能力标签、成本策略和错误归类。
- 自定义供应商不限制为固定槽位，产品层应支持动态新增、编辑、删除。
- 供应商配置与 Agent 配置解耦：供应商管理密钥、Base URL、协议和模型；Agent 管身份、提示词、技能、默认模型偏好和工具权限。

#### 推荐配置字段

- `provider_id`：供应商唯一 ID。
- `display_name`：用户可见名称。
- `enabled`：是否启用。
- `auth_type`：API Key、OAuth、本地网关或无需认证。
- `secret_ref`：本地安全存储引用，不保存明文密钥。
- `base_url`：官方或自定义接口地址。
- `api_format`：协议格式，例如 OpenAI-compatible、Anthropic Messages、Gemini。
- `models`：可用模型列表。
- `default_model`：默认模型。
- `capabilities`：结构化输出、工具调用、流式输出、视觉、多模态、长上下文、低成本等能力标签。
- `region`：区域或线路标签。
- `cost_policy`：价格、预算和估算方式。
- `rate_limit_policy`：限流和重试策略。
- `health_check`：连接测试和模型可用性检测规则。

#### 需求

- 用户可配置一个或多个供应商。
- 创建 AI 整理任务时可以选择供应商和模型。
- 自然语言智能采集时，AI 必须先把用户需求转成结构化采集计划。
- AI 采集计划必须包含平台、数据类型、国家/地区、关键词或账号、时间范围、预计调用步骤。
- 支持默认模型配置。
- 支持模型连接测试。
- 支持模型调用失败重试。
- 支持模型输出 JSON Schema 校验。
- 支持模型调用成本估算。
- 支持采集计划 Schema 校验；校验失败时不得调用 TikHub API。
- MVP 必须支持提示词版本管理和提示词优化；这不是 V1 或高级能力，而是自然语言采集和 AI 整理可靠性的基础能力。
- 提示词优化目标必须是稳定结构化输出、降低歧义、减少成本、保留证据引用、明确失败边界和减少数据 slop，不允许通过提示词绕过参数白名单、成本确认或人工确认。
- 支持把每次运行锁定为运行快照，记录实际使用的供应商、模型、协议格式、Base URL 类型、提示词版本和能力标签，避免全局设置变更后无法复现历史任务。

#### MVP 提示词优化范围

MVP 至少内置以下提示词优化能力：

- 采集计划提示词：把自然语言稳定转成平台、数据类型、国家/地区、关键词、账号、时间范围、调用步骤和成本估算所需字段。
- 追问策略提示词：当平台、数据类型、时间范围、预算条件缺失，或任务需要国家/地区但条件缺失时，模型必须追问，不能猜测执行。
- 结构化输出提示词：所有 AI 整理模板必须绑定 JSON Schema，并明确字段类型、可空规则、置信度和证据引用。
- 证据引用提示词：摘要、情绪、主题、实体、风险、代表性观点和爆点拆解必须引用原始记录 ID 或记录集合 ID。
- 失败边界提示词：当证据不足、字段冲突、输入过少、模型无法判断或输出不完整时，必须输出待人工确认或失败原因。
- 成本控制提示词：提示词应限制无必要长篇输出，优先生成可导出、可校验、可复用的结构化字段。
- 多供应商兼容提示词：提示词不得依赖某一家模型的私有行为，应通过 Schema、示例和校验规则提高跨供应商稳定性。

MVP 不要求做复杂的自动 prompt search 或在线 A/B 测试，但必须提供可维护的提示词模板、版本号、变更说明、回滚能力和基础回归样例。

#### AI 整理模板

- 自然语言采集计划生成。
- 通用摘要。
- 评论情绪分析。
- 主题分类。
- 实体抽取。
- 代表性观点提取。
- 内容爆点拆解。
- 风险与异常提示。
- 平台差异对比。

#### 结构化输出要求

每次 AI 运行必须记录：

- 任务 ID。
- 运行类型：采集计划生成或数据整理。
- 输入记录集合 ID。
- 供应商 ID。
- 模型 ID。
- 协议格式。
- Base URL 类型：官方、自定义、本地网关。
- 提示词版本。
- 输出 Schema 版本。
- 输入摘要。
- 输出结果。
- 校验状态。
- 运行时间。
- 失败原因。
- 成本估算。
- 输入 Token、输出 Token 或供应商返回的等价计量。
- 首字延迟、总耗时、重试次数。

#### 反数据 slop 门禁

本产品把“数据 slop”定义为：看起来已经整理完成，但缺少来源引用、字段级证据、Schema 校验、冲突说明、置信度、人工确认或失败边界的数据结果。

必须满足：

- AI 生成的采集计划必须可预览、可编辑、可拒绝、可追溯。
- AI 生成字段必须保存来源记录 ID、转换理由、置信度、校验状态、模型或规则版本、人工审核状态。
- AI 推断字段必须带置信度；凡涉及国家/地区及其标准化代码、语言、受众市场、情绪、主题、风险或意图等推断字段，均不得只保存结论。
- 洞察、摘要、主题和情绪结论必须能回到原始记录集合，不得只保存一段无法核验的自然语言结论。
- 状态必须区分草稿、等待确认、运行中、部分成功、成功、失败、已取消、待人工确认；不得把待确认或阻塞状态汇总成成功。
- 模型输出不符合 Schema、字段证据缺失、来源记录不存在、参数白名单失败时，必须进入失败或待人工确认状态。
- Excel 和 PDF 不输出未校验且无来源引用的核心结论。
- UI 和报告中禁止使用“全部无误”“已自动确认”“已处理所有问题”等无证据绝对化文案。

#### 验收标准

- 任一供应商可用时即可完成 AI 整理。
- 用户可以配置内置供应商，也可以新增自定义 OpenAI-compatible 供应商并完成连接测试。
- 用户自然语言描述数据需求后，AI 能生成可编辑的结构化采集计划。
- 采集计划未确认前，不会产生 TikHub 正式采集费用。
- 模型返回非 JSON 时，系统提示格式校验失败，并保留原始模型输出。
- AI 结果不能覆盖原始数据。
- AI 结果必须能追溯到输入记录。
- AI 采集计划必须能追溯到原始自然语言需求。
- 任何导出的核心 AI 洞察都能追溯到原始记录 ID、模型运行记录和提示词版本。
- 修改任一 MVP 提示词模板后，必须能用固定样例验证采集计划 Schema、整理结果 Schema、证据引用和失败边界仍然通过。

### 6.5 任务管理

#### 任务状态

- 草稿。
- 等待确认。
- 排队中。
- 运行中。
- 部分成功。
- 成功。
- 失败。
- 已取消。
- 待人工确认。

#### 需求

- 支持新建任务。
- 支持任务来源标记：表单式采集、自然语言智能采集。
- 支持自然语言采集计划的确认、修改和重新生成。
- 支持取消任务。
- 支持失败重试。
- 支持复制任务。
- 支持查看运行日志。
- 支持查看成本预估和实际成本。
- 计价账本保存 TikHub 报价快照和累计报价，并明确标记为报价而非最终账单；不得用虚假的零实际成本结算请求。
- 支持按平台、状态、时间筛选任务。
- 支持任务级国家/地区配置，并用于账号标记、账号筛选、直播候选校验和报告统计口径；标准化国家代码作为国家/地区的存储值或筛选值。

#### 成本确认

以下情况必须二次确认：

- 预计 TikHub 请求数超过用户设置阈值。
- 预计模型调用成本超过用户设置阈值。
- 单任务预计运行时间超过 10 分钟。
- 移动端任务被判断为长任务。
- 自然语言采集计划包含多个平台或多个数据类型。
- 直播状态监测、开播触发或公开互动来源账号发现预计请求量超过用户设置阈值。

#### 验收标准

- 用户能清楚看到任务当前状态。
- 用户能清楚看到任务是表单式创建还是自然语言创建。
- 自然语言任务必须先展示采集计划，再进入运行状态。
- 任务失败必须显示失败阶段和错误说明。
- 取消任务后不再发起新的外部请求。
- 部分成功任务可以导出已完成数据。

### 6.8 导出与自动化

#### 导出格式

MVP 对外导出格式：

- Excel 工作簿（.xlsx）。
- PDF 报告。

账号数据 Excel 必须使用`社交平台用户数据收集模板_国家地区版.xlsx`定义的结构，并满足：

- 只保留`用户数据收集表`、`填写说明`、`字段枚举`和`资料依据`四个工作表，不增加原始数据附表。
- `用户数据收集表`严格保持 18 列顺序，年龄写入数值列，未知年龄留空。
- 至少保留模板原有 200 行数据区；超过 200 条时扩展样式、公式与数据校验。
- 地区来源、地区置信度、平台、性别、年龄和数据来源的数据校验分别作用于 G、H、I、J、K、P 列。
- 原始响应、报价快照和逐目标失败详情保存在本地证据库与运行日志，不写入额外工作表。

不把 CSV、JSON、JSONL、Markdown 作为 MVP 对外主导出格式。若开发调试或 n8n 自动化确实需要结构化文本数据，优先通过 Webhook 摘要解决；完整文件格式可放到 V1 作为高级导出。

#### Webhook

MVP 支持任务完成后向用户配置的 Webhook URL 发送摘要数据。

Webhook 内容包括：

- 任务 ID。
- 任务名称。
- 平台。
- 状态。
- 数据量。
- 核心摘要。
- 导出文件路径或用户可访问的本地路径说明。
- 生成时间。

由于 MVP 是本地优先产品，Webhook 只在本机任务完成时触发，不保证离线补发。

#### n8n 处理

- 不在主导航突出 n8n。
- 在导出或自动化设置中提供 Webhook 说明。
- 帮助文档提供 n8n 接入示例。
- V1 可提供 n8n 工作流模板。

#### 验收标准

- 用户可以导出 Excel 工作簿和 PDF 报告。
- Excel 工作簿中文不乱码，四工作表与 18 列结构完整，200、201 和 1,200 条边界下公式、样式与校验范围正确。
- PDF 报告中文不乱码，证据引用完整。
- Webhook 成功和失败都有日志。
- Webhook 不发送 API Key。

## 7. 技术栈指南

### 7.1 总体架构

MVP 采用本地优先 macOS 桌面应用架构：

```text
macOS 客户端
  |
  | 本地保存 API Key
  | 本地发起 TikHub / 模型供应商请求
  | 本地任务队列、缓存、报告生成
  v
本地数据库 / 本地文件存储
  |
  v
导出 Excel 工作簿 / PDF 报告
```

MVP 不设置自有云端 API 代理，不接远端数据库，不做云端任务或云端密钥托管。自有服务器不保存用户密钥和原始数据，密钥保存在本机系统安全存储，原始数据按工作区策略保存在本机。

### 7.2 前端与 macOS 壳

前端负责 macOS 工作台、表单、任务状态、数据查看、证据追溯和导出状态，不负责直接访问外部 API、系统安全存储或本地数据库。具体库、组件边界和状态策略以《前端技术方案》为单一事实源。

### 7.3 本地运行与数据层

推荐技术：

- SQLite 作为本地数据库。
- 本地文件系统保存原始响应、Excel 文件、PDF 文件。
- 系统安全存储保存 API Key。
- 本地队列管理任务状态。

数据分层：

- 结构化索引数据存 SQLite。
- 大体积原始响应存文件。
- 表格结果存 Excel 文件，报告结果存 PDF 文件。
- 敏感密钥不进入 SQLite。

### 7.4 外部 API 调用

调用对象：

- TikHub REST API。
- 内置模型供应商 API：OpenAI、Gemini、Claude。
- 扩展模型供应商 API：DeepSeek、Qwen、Moonshot、OpenRouter 等。
- 本地模型网关：Ollama 或用户自定义本地 endpoint。
- 自定义 OpenAI-compatible 接口。
- 后续可接 TikHub MCP。

统一请求层必须提供：

- 超时。
- 重试。
- 限流识别。
- 错误归类。
- 成本记录。
- 请求日志脱敏。
- 响应结构校验。

#### 模型适配层

模型适配层负责把不同供应商和协议格式统一成应用内部可控的调用能力。

必须提供：

- Provider Registry：集中维护供应商 ID、显示名称、默认 Base URL、协议格式、默认模型、能力标签、区域和成本策略。
- Resolver：根据任务选择、默认模型和供应商启用状态，解析出一次运行实际使用的供应商、模型、Base URL、协议格式和密钥引用。
- Adapter：适配 OpenAI-compatible、Anthropic Messages、Gemini 等协议差异，包括结构化输出、工具调用、流式输出和错误格式。
- Runtime Snapshot：每次 AI 运行锁定当时的供应商、模型、协议、提示词版本和配置来源，保证历史结果可复现。
- Connection Test：每个供应商必须能做低成本连接测试；测试失败时给出供应商、协议、状态码或网络错误原因。
- Redaction：请求日志、错误日志、运行快照和导出文件不得包含完整 API Key。

V1 可考虑增加本机 CLI 配置复用能力，例如读取用户已配置的 Claude Code、Codex、Qwen Code 等本机配置，但必须以用户显式授权为前提，且不得污染用户原有配置。

### 7.5 双入口调用架构

#### 表单式精确采集

表单式采集由应用直接生成 TikHub API 请求：

```text
表单参数
  |
  v
参数校验
  |
  v
成本估算
  |
  v
用户确认
  |
  v
应用调用 TikHub API
```

要求：

- 平台、数据类型、国家/地区、关键词、账号、时间范围等参数由用户显式选择或输入。
- 应用根据平台和数据类型映射到 TikHub API。
- 应用负责注入本地保存的 TikHub API Key。
- AI 不参与该模式的 API 选择和调用。

#### 自然语言智能采集

自然语言采集由 AI 生成结构化计划，应用负责校验和执行：

```text
自然语言需求
  |
  v
AI 生成采集计划
  |
  v
应用校验计划和参数白名单
  |
  v
应用展示计划与成本
  |
  v
用户确认
  |
  v
应用按计划调用 TikHub REST API
  |
  v
应用保存请求记录和响应摘要
```

要求：

- AI 不直接持有 TikHub API Key。
- AI 不直接发起外部网络请求。
- MVP 不让 AI 通过 MCP 或工具链直接调用 TikHub。
- 应用根据采集计划映射到内部 TikHub REST API 请求，例如搜索内容、获取详情、采集评论、获取用户公开信息。
- 每类内部请求都必须绑定平台、数据类型、国家/地区、参数 Schema、调用上限和成本上限。
- 应用必须记录 AI 生成的计划、最终 TikHub API 请求和响应摘要。
- 用户确认前，只允许做免费或低成本的连接测试、参数校验和成本估算。

### 7.6 TikHub API 与 MCP 策略

MVP：

- 优先接 TikHub REST API。
- 用确定性接口完成采集。
- 自然语言采集使用“AI 生成计划 + 应用直接调用 TikHub REST API”的方式实现。
- 暂不把 MCP 作为核心执行路径。

V1：

- 接入 TikHub MCP Streamable HTTP。
- 支持模型通过 MCP 做代理式探索，但作为高级 Agent 模式单独开启，仍必须经过权限、成本和参数白名单约束。
- 记录 MCP 工具名、入参、出参和费用。

策略原因：

- REST API 更适合确定性采集、错误处理和成本控制，也更适合 MVP 的自然语言计划执行。
- MCP 更适合 AI 代理探索，但更需要审计和安全边界。

### 7.7 Excel 技术方案

Excel 由 Rust 后端导出器生成，前端只触发任务并展示状态和结果路径。具体实现以《技术框架总览》和《后端技术方案》为准。

用途：

- 生成 `.xlsx` 工作簿。
- 创建多工作表。
- 设置列宽、表头、冻结行、筛选和基础样式。
- 写入超链接。
- 写入大数据量时使用流式写入能力。

关键工作表：

- `任务概览`：平台、条件、时间、状态、成本、模型。
- `原始数据摘要`：原始记录 ID、平台、原始链接、采集时间、状态。
- `标准化数据`：统一字段后的内容、作者、互动数、发布时间、标签。
- `AI 结构化结果`：摘要、情绪、主题、实体、风险、置信度。
- `洞察与证据`：洞察、证据记录 ID、原始链接、模型、提示词版本。
- `运行日志`：阶段、时间、状态、错误摘要。
- `成本明细`：API 请求量、模型成本、有效入库成本、过滤损耗、估算费用。

验收要求：

- Excel 文件可被 Microsoft Excel、Apple Numbers、WPS 和 LibreOffice 打开。
- 中文、英文、链接、长文本显示正常。
- 表头固定，关键列宽可读。
- 不包含明文 API Key。
- 大任务导出时不能让应用明显卡死。

### 7.8 PDF 技术方案

PDF 由 Rust 后端导出器生成，前端负责预览摘要、触发导出并展示状态和结果路径。具体实现以《技术框架总览》和《后端技术方案》为准。

关键实现原则：

- 报告数据模型与渲染层分离。
- Excel 和 PDF 共用同一份报告数据。
- PDF 组件不直接读取任务数据库。
- PDF 生成前先进行字体检查。
- 生成完成后校验文件大小、页数、标题和导出路径。

### 7.9 数据模型

核心实体：

- workspace：本地工作区。
- secret_ref：本地密钥引用，不保存明文。
- connector：外部连接配置。
- model_provider：模型供应商配置。
- model_profile：模型配置和能力标签。
- agent_profile：Agent 身份、系统提示词、技能和默认模型偏好。
- runtime_snapshot：一次 AI 运行实际使用的供应商、模型、协议、提示词和配置来源快照。
- collection_task：采集任务。
- task_intent：自然语言需求原文和解析状态。
- collection_plan：AI 生成的结构化采集计划。
- api_call_step：计划中的单个受控 API 调用步骤。
- task_run：任务运行记录。
- raw_record：原始记录索引。
- normalized_record：标准化记录。
- prompt_template：AI 整理和采集计划生成模板。
- prompt_version：提示词版本。
- ai_run：AI 运行记录。
- field_provenance：字段级来源证据、转换理由、置信度和校验状态。
- insight：洞察结果。
- report：报告记录。
- export_job：导出任务。
- webhook_job：Webhook 任务。
- audit_log：审计日志。

位置类字段标准：

- 位置字段统一命名为“国家/地区”，不得在产品界面、导出表头或内部业务字段中混用其他同义字段名。
- 所有位置类字段必须包含“来源”和“置信度”；来源可来自用户输入、TikHub 原始字段、账号公开资料、内容公开元数据、直播公开元数据、AI 推断或人工修正。
- 国家/地区字段必须保留原始值、标准化国家代码、来源、置信度和校验状态；低置信度或来源冲突时进入待人工确认。
- AI 推断出的国家/地区及其标准化代码、语言、受众市场等位置相关字段必须带置信度和证据引用，不得作为无来源硬过滤条件。

关键关系：

```text
collection_task
  -> model_provider
  -> model_profile
  -> agent_profile
  -> task_intent
  -> collection_plan
  -> api_call_step
  -> task_run
  -> raw_record
  -> normalized_record
  -> prompt_version
  -> runtime_snapshot
  -> ai_run
  -> field_provenance
  -> insight
  -> report
  -> export_job
```

### 7.10 本地备份策略

MVP 必须支持手动备份。

备份内容：

- 工作区配置。
- 任务元数据。
- 标准化数据。
- AI 结果。
- 报告数据。
- 导出索引。

默认不备份：

- API Key。
- 明文请求 Header。
- 临时缓存。

用户可选择是否包含原始响应文件。

若未来支持密钥备份，必须使用用户设置的本地密码进行加密，并作为独立安全需求评审。

## 9. 安全、隐私与合规

### 9.1 安全要求

- API Key 只保存在本地安全存储中。
- 日志中不得输出完整 API Key。
- 导出文件中不得包含 API Key。
- PDF 中不得包含 API Key。
- Webhook 中不得包含 API Key。
- 删除密钥后，依赖该密钥的任务不可继续运行。
- 所有外部请求必须使用 HTTPS。

### 9.2 隐私要求

- 默认不上传用户任务数据到自有服务器。
- 默认不上传原始社交数据到自有服务器。
- 默认不上传 AI 结果到自有服务器。
- 崩溃日志必须脱敏。
- 匿名使用统计必须明确告知用户，并允许关闭。

### 9.3 合规边界

- 只采集公开数据。
- 不提供绕过登录、验证码、权限限制的能力。
- 不承诺用户对数据的二次使用天然合规。
- 报告中必须包含 AI 免责声明和数据来源声明。
- 用户应对导出、传播和后续处理承担责任。

### 9.4 免责声明建议

PDF 报告默认包含以下声明：

```text
本报告由智能数据整理平台基于用户配置的公开数据采集任务和 AI 模型生成。AI 生成内容可能存在遗漏、误判或表达偏差，使用前应由人工复核。报告中的社交平台数据来源于用户授权调用的第三方数据服务，用户应自行确认后续存储、传播和使用行为符合适用法律法规及平台规则。
```

## 12. 风险与应对

### 12.1 本地优先限制

风险：

- 换设备后无法自动同步。
- 用户卸载应用可能丢失数据。

应对：

- 提供本地备份导出。
- 卸载或清空数据前强提醒。

### 12.2 外部 API 不稳定

风险：

- TikHub 或用户选择的模型供应商不可用会影响任务。
- 自定义模型供应商可能存在协议兼容差异。

应对：

- 任务分阶段保存。
- 支持部分成功。
- 支持失败重试。
- 错误信息归类。
- 保留已采集数据。
- 模型适配层必须区分供应商错误、协议错误、Schema 错误和网络错误。

### 12.3 成本失控

风险：

- 大规模采集和多模型分析会快速增加费用。

应对：

- 运行前成本预估。
- 用户自定义单任务上限。
- 用户自定义每日上限。
- 高成本任务二次确认。

### 12.4 AI 可信度

风险：

- AI 可能误判情绪、事实、主题或趋势。
- AI 可能生成看似完整但缺少来源、证据和失败边界的数据 slop。

应对：

- 所有洞察绑定来源证据。
- 显示模型和提示词版本。
- 支持人工确认。
- 支持保留原始模型输出。
- 字段级保存来源记录、转换理由、置信度和校验状态。
- 未通过 Schema 校验或缺少证据的内容不得进入核心导出。

### 12.5 PDF 复杂度

风险：

- 中文字体、分页、表格和图表可能导致 PDF 渲染问题。

应对：

- MVP 限制 PDF 版式复杂度。
- 图表可先导出为静态图片或摘要表。
- PDF 生成后做文件完整性校验。
- 字体缺失时阻止生成并提示。

## 13. 数据与接口边界

### 13.1 不上传自有服务器的数据

MVP 不接远端数据库，也不上传以下数据到自有服务器：

- API Key。
- 原始社交数据。
- 标准化数据。
- AI 输出。
- 报告文件。
- 任务日志。

### 13.2 会发送给外部供应商的数据

用户运行任务时，应用会直接从本机向以下服务发送必要数据：

- TikHub：发送采集参数和 TikHub API Key。
- 用户选择的模型供应商：发送用户选择的记录内容和对应模型供应商 API Key。
- 自定义模型 endpoint：发送用户选择的记录内容、Base URL 请求和对应密钥。
- Webhook 目标：发送用户启用的任务摘要，不发送密钥。

### 13.3 用户可控项

- 是否运行采集。
- 是否运行 AI 整理。
- 选择哪个模型供应商。
- 选择哪些字段发送给 AI。
- 是否导出报告。
- 是否启用 Webhook。
- 是否保留原始响应。

## 15. 最终定版摘要

MVP 要构建的是一个本地优先的 macOS 智能数据整理平台。用户在本机配置 TikHub 和模型供应商密钥，通过 TikHub 采集 TikTok、抖音、小红书三个平台的公开数据，再用用户选择的模型完成摘要、分类、情绪、实体和洞察生成，最终导出 Excel 表格和 PDF 报告。

MVP 不做账号体系、不接远端数据库、不做远端同步、不做多设备自动同步、不做团队协作、不做云端任务、不做云端密钥托管、不做自研爬虫。模型接入不限制为 OpenAI、Gemini、Claude 三家，产品应以 Provider Registry、协议适配层和运行快照支持更多供应商与自定义 OpenAI-compatible endpoint。n8n 不作为核心卖点，但通过本机 Webhook 和文件导出保留接入路径。产品最重要的差异化不是“自动生成结论”，而是“本地可控、成本清楚、来源可追溯、报告可交付、拒绝数据 slop”。
